How to read this diagram

Top-down flow (the request path)

  1. User clicks something on a Vaadin FooView (or a REST client calls a controller).
  2. The matching FooPresenter (or controller) calls a single Application service method.
  3. The service may call 1 or more repository interfaces (defined in the Domain) and/or application ports (e.g. payment).
  4. At the bottom, Spring DI wires the repository interfaces to their Memory*Repository implementations and the ports to their concrete adapters (Bcrypt, Jwt, Stub*, Vaadin*).
  5. Each Memory*Repository uses RepositoryLocks for per-key concurrency. LoggingAspect wraps every service method.

Bottom-up flow (the implementation arrow direction)

Dashed lines with an open triangle at the upper end mean "implements / realizes". Every Memory*Repository realizes its IXxxRepository; every adapter (JwtSessionManager, BcryptPasswordHasher, etc.) realizes its application-layer port. All IXxxRepository interfaces extend the shared IRepository<T, ID> contract in the cross-cutting bar.

Inside an aggregate frame

The yellow box on the left is the «Aggregate root» repository — the only way external code reaches the aggregate. The white box with the thick green border (next to it) is the root entity. Solid lines with a filled diamond at the root represent composition (sub-entities and VOs owned by the root). Multiplicities are written near each end.

Cross-aggregate references

The very thin grey dashed lines between aggregate frames represent fields like companyId on CompanyAppointment, recipientUserId on Notification, buyerUserId on OrderReceipt, etc. Per the course rule "no aggregate-to-aggregate diagram relations", these are not drawn on the official deliverable — they exist in the code as ID fields. They're shown here for teaching only.
