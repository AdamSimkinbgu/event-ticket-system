Why four diagrams instead of one

A single canvas with every box and every arrow becomes unreadable — 50+ arrows crossing 80+ boxes can't be auto-routed cleanly, and hand-routing SVG line segments hits the obstacle problem you saw. The fix is to split by concern. Each diagram below has fewer than ~50 nodes, which lets Mermaid's layout engine produce arrows that don't cross boxes.

Together the four cover the same scope as the single combined diagram: every layer, every aggregate's internal structure, every service-to-repository dependency, and every realization. Just split so each one is legible.

1. Presentation → Application — request flow

How a user click reaches a service. Vaadin views pair 1:1 with presenters (MVP), and the presenter is the only thing that calls the service. REST controllers do the same on the machine-facing side.

Per Tier C there's no suspension view (II.6.7–9 exempt), no real-time push (notification bell refreshes on view-load / login flush), and no discount-policy editor — only the purchase-policy editor.

4. Infrastructure realizations — who implements what

Every IXxxRepository (Domain) has a single Memory*Repository implementation (Infrastructure), and every external port (Application) has a concrete adapter. Solid arrows with an open triangle = "implements / realizes".

Every Memory*Repository also uses RepositoryLocks for per-key concurrency. LoggingAspect wraps every @Service method. Both are shown at the bottom of the diagram.

How to use these diagrams

Read top-to-bottom for a request
