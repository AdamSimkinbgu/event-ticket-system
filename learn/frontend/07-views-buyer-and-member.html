§ 2landingThe first page anyone sees

LandingView — route: / · MainLayout · @AnonymousAllowed

The site root. Designed to convert guests into signed-in members and route members straight to where they want to go. Conditional hero based on whether the user is signed in.

What it shows

  • Gradient hero with primary CTAs — signed-out gets "Sign up free" + "Browse without signing up"; signed-in gets "Browse events" + "My tickets"
  • Category tile row — concerts / sports / theatre / conferences. Each tile deep-links to BrowseEventsView?cat=…
  • "Featured this week" row of BzPoster cards
  • "On sale soon" upcoming-events strip

The conditional hero

if (MockAuth.isSignedIn()) {
    heroCta.add(new LkBtn("Browse events").variant(LkBtn.Variant.primary).onClick(…));
} else {
    heroCta.add(
        new LkBtn("Sign up free").variant(LkBtn.Variant.primary).onClick(…),
        new LkBtn("Browse without signing up").variant(LkBtn.Variant.tertiary).onClick(…)
    );
}

This is one of the rare places we read MockAuth.isSignedIn() directly instead of going through Capabilities. Justified because "is the user signed in" is an identity question (state), not an action question (capability).

§ 3catalogBrowse, drill in, pick seats

BrowseEventsView — route: browse · MainLayout · @AnonymousAllowed

The catalog grid. Filter chips along the top (Category, Region, Sort, Price) + a card-grid of event posters below. Built almost entirely from kit primitives.

What it composes

Kit classUsed for
LkFilterChipThe 4 filter chips. Multi-select for category/region, single-select for sort.
LkSelectThe sort picker, label "Sort by".
BzPosterEach event card — gradient block + category chip + title + meta + price.

The category deep-link from the landing page

URL browse?cat=music auto-applies the "Music" filter on entry. The view reads BeforeEnterEvent.getLocation().getQueryParameters() inside its constructor and calls setApplied(List.of("Music")) on the category chip. Wins free for deep-linkable filters.

EventDetailsView — route: event/:eventId · MainLayout · @AnonymousAllowed

Drill-in screen for one event. Header with poster + title + datetime, body with description + zone preview, CTA to reserve seats. Compact VkVenueMap embedded as a clickable preview.

Notable bits

  • Route parameter via :eventId. Currently ignored (mock shows Coldplay regardless); real wiring is on #272 V2-WIRE-BUYER-EVENT.
  • Uses VkVenueMap in compact mode — smaller floor plan, no entrances row.
  • "See seats" button navigates to SeatPickerView with the same eventId.

SeatPickerView — route: event/:eventId/seats · MainLayout · @AnonymousAllowed

The centerpiece of the buyer flow. Full-size VkVenueMap on the left, zone detail panel on the right with either a VkSeatBlock (for seated zones) or a VkStandingZone + quantity stepper.

What happens on "Add to cart"

MockCart.add(new MockCart.Item(
    "Coldplay · MOTS",           // event title
    "VIP · A12",                 // seat label
    32000,                          // $320 in cents
    Instant.now().plus(Duration.ofMinutes(10)),  // 10-min hold
    "linear-gradient(135deg,#8b5cf6,#ec4899)" // concert gradient
));

Once added, the navbar cart badge increments, the timer chip starts ticking down, and the user is forwarded to CartView.

§ 4orderThe cart → checkout → confirmation flow

CartView — route: cart · MainLayout · @PermitAll

Per-line reservation rows with a live countdown chip on each. "Proceed to checkout" CTA at the bottom enters CheckoutView and triggers the global deadline.

The countdown chip

Each line gets a .bz-timer chip backed by a one-line executeJs that sets up a setTimeout chain on the client. The server only ships the endMs once; everything else is browser-side. Counter turns red (urgent) under 3 minutes; chip removes itself on hit-zero.

Empty state

When the cart is empty the view shows "Your cart is empty" with a "Browse events" button. The view detects this via MockCart.getItems().isEmpty() and renders an empty-state card instead of the grid.

CheckoutView — route: checkout · MainLayout · @PermitAll

The pay screen. Payment form on the left (card number, expiry, CVC, save-this-card) + line summary on the right. Single global countdown banner at the top instead of per-line chips.

The global deadline trick

if (MockCart.getCheckoutDeadline() == null) {
    MockCart.setCheckoutDeadline(Instant.now().plus(Duration.ofMinutes(10)));
}
Instant deadline = MockCart.getCheckoutDeadline();

On first entry, set the global deadline. On re-entry (user navigated away and back), reuse the existing one. The banner countdown reads from this shared value. The navbar timer chip in MainLayout also picks this up via the shortest-deadline union.

Submit handler

"Pay now" currently fires a Toasts.success and navigates to OrderConfirmationView with a mock order id. The cart is cleared. Real wiring lives in #282 V2-WIRE-CHECKOUT-PAY.

OrderConfirmationView — route: order/:orderId · MainLayout · @PermitAll

The "done!" screen. Big green check, order id, line-item recap, total paid, two CTAs (View my tickets / Email receipt) and a small printable hint.

Currently uses hardcoded mock data — real wiring (look up the order by id) is part of #222 V2-CHECK-02's post-merge follow-up.

§ 5accountThe member account surfaces

MyAccountView — route: my-account · MainLayout · @PermitAll

The hub. Two tabs (Tickets / Orders) inside one card. Each tab renders an LkGrid of mock rows. Clicking a ticket row opens the BzTicketDialog ticket-stub modal; clicking an order navigates to ReceiptView.

How the tabs work

Uses Vaadin's built-in Tabs + TabSheet. Each tab's content is wrapped in a Div and added to the tab sheet. Tabs is one of the rare native Vaadin components used here — the kit doesn't have its own tab primitive (and doesn't need one when Lumo's is fine).

MyProfileView — route: my-profile · MainLayout · @PermitAll

Read-only profile display. Name, email, registration date, account-type badge. No edit form — II.3.4 is exempt per the Tier C accommodation, so a real "edit profile" surface was never built. The view's comment block notes this explicitly.

MyInvitationsView — route: my-invitations · MainLayout · @PermitAll

Pending manager and co-owner invitations the user has received. Card per invitation with Accept / Decline buttons (currently toast-only; real wiring on #275 V2-WIRE-MEMBER-INVITES).

Why it's not in the avatar menu directly

It is — the menu's "My invitations" item navigates here. The view itself is just a list. Empty state ("No pending invitations") when there are none.

ReceiptView — route: receipt/:orderId · MainLayout · @PermitAll

A printable receipt page for one order. Same shape as OrderConfirmationView but designed for window.print() — minimal chrome, clear line items, big barcode at the bottom.

The "Request refund" button

If the order is refund-eligible (mock check), shows a "Request refund" button at the bottom. Clicking opens the BzRefundDialog modal. The dialog calls back with a Toasts.success — real wiring on #284 V2-WIRE-MEMBER-REFUND.

SupportInboxView — route: support · MainLayout · @PermitAll

Master/detail inbox for the user's outgoing complaints. Same shape as CompanyInquiryInboxView in 08 — left card with MdConvRows, right card with MdThread + MdReplyBar.

The first place a teammate touching messaging should look — everything they need to build the company-side inbox is already here in member shape.

§ 6authSign in & register

LoginView — route: login · MainLayout · @AnonymousAllowed

The unified sign-in surface (covered in 03 §11). One form, decides between MockAuth.signIn and MockAuth.signInAsAdmin based on whether the username is in MockAuth.ADMIN_USERNAMES. Routes to BrowseEventsView or AdminDashboardView accordingly.

RegisterView — route: register · MainLayout · @AnonymousAllowed

Signup form. Username + email + password + confirm + Terms checkbox. Submit checks password-mismatch + length, refuses admin usernames (the disjoint-pool rule), calls MockAuth.signIn(username), navigates to BrowseEventsView.

The admin-name refusal

if (MockAuth.isAdminUsername(username.getValue())) {
    Toasts.failure("That username is reserved for the platform-admin pool.");
    return;
}

This is the other half of the disjoint-pool enforcement (the first half lives in LoginView). Together they guarantee that the same name can never resolve to both a member and an admin.

§ 7messagingOne-off complaint composer

SubmitComplaintView — route: support/complaint · MainLayout · @PermitAll

A focused one-screen form — subject (TextField), body (TextArea), category (LkSelect mock), optional order reference. Submit toasts and forwards back to SupportInboxView. Real wiring on #267 V2-WIRE-MSG-COMPLAINT.

Why this is its own view (not a dialog)

The original wireframe had it as a modal. We made it a routed view because: (1) deep-linkable — a help link in an email could land users here directly, (2) the form is large enough that a dialog felt cramped, (3) the user's session might expire mid-compose, and a routed view's draft can be restored on reload while a dialog's is gone.

§ 8Common patterns across the tour

Pattern 1 — LkPage as the spine

Every view extends LkPage. The constructor calls title(…), optionally subtitle(…) + actions(…), then add(…) for the body. Reading any view's constructor top-to-bottom tells you exactly what the page renders, in DOM order.

Pattern 2 — Filter chips + grid card

BrowseEventsView, MyAccountView (orders tab) all share this shape: LkRow of LkFilterChips on top, LkCard.pad(0) wrapping a LkGrid below. Three lines of Java per filter chip, ~15 lines per grid column.

Pattern 3 — Toast-and-navigate submit

Submit handlers throughout this tier follow the same shape: 

submitBtn.onClick(e -> {
    if (form.invalid()) {
        Toasts.failure("Fix the highlighted fields.");
        return;
    }
    // mutate session / call mock service
    Toasts.success("Done!");
    UI.getCurrent().navigate(NextView.class);
});

When V2-ERROR-01 lands, the Toasts.failure(String) call becomes Toasts.failure(ErrorPayload) with structured failure data. Every submit handler on the tier inherits the change for free.

Pattern 4 — Empty states are first-class

CartView, MyAccountView, MyInvitationsView, SupportInboxView all check their data source for emptiness before rendering the grid. The empty state is a centred card with an icon, a one-sentence explanation, and a CTA to "do the thing that would populate this list."

If you skip empty states, the view shows headers + a blank grid + nothing else. Users assume the app is broken. The empty-state card is the cheapest UX investment in the entire tour — ~15 lines of Java per view, immediate clarity.

Pattern 5 — Identity checks via MockAuth, action checks via Capabilities

The split:

  • MockAuth.isSignedIn() — reads whether the user is in a session. Used for the landing-page hero choice, the avatar-vs-guest-actions topbar slot, the empty-state messages ("Sign in to see your tickets").
  • Capabilities.has(X) — reads whether the user can do X. Used for buttons, sections, drawer items.

Both are legitimate. The first is about state, the second about authority. Mixing them up ("use Capabilities for everything") leads to weirdness like Capabilities.has(CHECKOUT) being your guest check.
