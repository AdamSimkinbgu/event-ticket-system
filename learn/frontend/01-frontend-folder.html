learn / frontend / 01

The frontend/ folder

Vaadin in 60 seconds · what generated/ is · then a deep dive into theme.json and styles.css.

§ 1What is Vaadin (Flow)?

Vaadin is a Java web framework where you write the UI as Java classes and Vaadin takes care of the browser side. You never write a fetch() call or set up a React component — you create a new Button("Submit") in Java and Vaadin renders it in the browser, listens to clicks, and ships them back to a Java event handler on the server.

Concretely:

  • Each view is a Java class (e.g. BrowseEventsView).
  • Each component is a Java class (TextField, Button, Dialog, our own LkCard…).
  • Each layout is a Java class (MainLayout, WorkspaceLayout).
  • The browser only loads a thin runtime (the “client engine”) plus our CSS.

So why is there a frontend/ folder at all? Because the runtime, theme, and any hand-written CSS still need to live somewhere — that’s what this folder is for.

Mental model. Java in src/main/java/ defines structure and behavior. src/main/frontend/ defines look and theme. Vaadin glues them together.

§ 2Folder layout at a glance

src/main/frontend/
├── index.html                # SPA shell — one <div id="outlet"> that Vaadin paints into.
├── policy-tree-map.js        # Hand-written D3.js module — Vaadin's @JsModule loads it to render
│                             # the policy editor's node-link tree (pan/zoom + inline +R/+G pills).
├── generated/                # Auto-built by Vaadin on every dev/build. Do NOT edit.
│   ├── flow/                 #   Vaadin's own runtime files.
│   ├── jar-resources/        #   CSS / JS extracted from Vaadin JARs at build time.
│   ├── theme.js              #   Auto-wires our theme into Vaadin's loader.
│   ├── theme.d.ts            #   Type defs for the above (only matters if you use TS).
│   ├── vaadin.ts             #   The client-side entry point Vaadin imports.
│   └── vaadin-featureflags.js # Toggles for opt-in Vaadin features.
└── themes/tickethub/        # OUR theme. The two files we own.
    ├── theme.json            #   Tells Vaadin which Lumo styles to pull in.
    └── styles.css            #   ~700 lines of our hand-written CSS.

Three files matter to a developer: theme.json + styles.css (the theme), and policy-tree-map.js (the one hand-written JS module — covered in 02-components.html under PolicyTreeMap). Everything else is auto-generated or one-line plumbing.

Why a custom JS module at all? Vaadin Flow is overwhelmingly Java-first — 99% of our UI is built with Java components. But when you need something the kit can't do (in this case, a D3.js node-link diagram with proper layout + pan/zoom), the pattern is: hand-write a small *.js file here, annotate the Java wrapper with @JsModule("./your-file.js"), and bridge events via @ClientCallable. Vaadin's Vite bundler picks up the file automatically. The current count of hand-written JS modules: 1.

§ 3generated/ — what is it?

When you run ./mvnw spring-boot:run in dev mode, or ./mvnw package in production mode, Vaadin walks the classpath, finds every @Route, @CssImport, and theme reference, and writes everything it needs into frontend/generated/.

File / folderWhat it holdsEdit?
vaadin.tsThe bundler entry point. Imports the Flow runtime + your theme.No
theme.jsWires tickethub theme into Vaadin's loader.No
flow/Vaadin's own client engine (server-to-browser bridge).No
jar-resources/CSS & JS extracted from Vaadin JARs (Lumo, components).No
vaadin-featureflags.jsFeature toggles. Set via vaadin.featureFlags.* in application.yml.No
Rule of thumb. If you edit anything inside generated/, your change disappears the next time Vaadin builds. generated/ is a build output, not source. Add it to .gitignore (it already is in this repo). If something looks wrong, blow it away — rm -rf src/main/frontend/generated — and let Vaadin re-create it.

The one file that lives alongside generated/ but isn’t generated is index.html:

src/main/frontend/index.html 
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
  <style>
    html, body, #outlet { height: 100%; width: 100%; margin: 0; }
  </style>
</head>
<body>
  <div id="outlet"></div>
</body>
</html>

That’s the whole single-page app shell. Vaadin mounts every view inside #outlet. You can add fonts, analytics tags, or favicons here — but anything UI-related belongs in styles.css.

§ 4Deep dive: theme.json

This file is exactly two lines today:

src/main/frontend/themes/tickethub/theme.json 
{
  "lumoImports": ["typography", "color", "spacing", "badge", "utility"]
}

What “Lumo” is

Lumo is Vaadin’s default design system — the set of CSS custom properties (--lumo-primary-color, --lumo-font-family, …) and base styles that every Vaadin component reaches for. You don’t have to use Lumo, but every vaadin-button, vaadin-text-field, vaadin-dialog etc. expects its tokens to exist.

lumoImports lets you cherry-pick which Lumo modules to include. Each module is a separately importable CSS file shipped inside the Vaadin Lumo JAR.

ModuleWhat it brings inRemove and you’ll lose…
typographyFont family, font sizes, headings, line heights.All Vaadin text scaling — everything becomes browser-default.
colorLumo color palette: --lumo-primary-*, --lumo-error-*, etc.Vaadin components render with no color theme — buttons go white-on-white.
spacing--lumo-space-xs/s/m/l/xl tokens used by every component’s padding/margin.Components collapse to default padding (very tight).
badgeBuilt-in span.badge styles (theme="badge primary", etc.).Our own LkBadge still works; only the built-in helper goes away.
utilityUtility classes like .flex, .gap-m, .text-center.Any built-in utility class on our views (we mostly use our own kit, so impact is small).

Other modules you could opt in to

ModuleUseful when
styleYou want Lumo’s opinionated base styles for body, lists, scrollbars.
presets/compactDensity preset — tightens spacing/typography for power users.

EXERCISE 1Force the “compact” density

Add "presets/compact" to lumoImports, run the app, navigate to /admin/sign-in, and notice how every Vaadin field gets shorter. 

{
  "lumoImports": ["typography", "color", "spacing", "badge", "utility", "presets/compact"]
}

Then remove it again. Watch the spacing return.

EXERCISE 2What breaks without color?

Temporarily delete "color" from the array. Restart. Open /login. The vaadin-button primary will look broken — it has no color tokens. That’s exactly what color brings in.

Lesson. Our kit components (LkBtn, LkCard) don’t depend on Lumo because they use our own --primary token from styles.css. But every native vaadin-* component does.
Where the tokens come from. Lumo’s actual CSS lives inside ~/.m2/repository/com/vaadin/vaadin-lumo-theme-*/lumo-theme.jar. The generated/jar-resources/ folder is where Vaadin extracts the bits you import. You can pop into jar-resources/ and read the real CSS to learn how Lumo is structured.

Naming conventions

Class names follow a 2-letter prefix that tells you which lane the style belongs to:

PrefixLaneExample
.lk-*Kit (shared design system primitives).lk-btn, .lk-card, .lk-topbar
.bz-*Buyer zone (catalog, cart, checkout).bz-timer, .bz-link
.vk-*Venue kit (seat picker, venue map).vk-seat, .vk-zone
.pe-*Policy editor shell (tree itself is now SVG via D3).pe-toolbar, .pe-plain, .pe-canvas, .pe-actionbar
.ow-*Owner workspace.ow-stats, .ow-tiles
.oc-*Org chart (admin).oc-card, .oc-av-founder
.md-*Master/detail (inbox).md-split, .md-detail-actions
Why the prefixes? CSS has a single global namespace. Prefixing prevents accidental collisions and makes it obvious which Java class is the owner. When you see .ow-edit-split in CSS, you know to look in views/company/EventManagementView.java.

Theme variants

Some classes change appearance via a modifier class on a parent. The clearest example is the admin platform shell: 

/* default blue topbar */
.lk-topbar { background: var(--primary); color: #fff; }

/* orange platform variant — added by PlatformAdminLayout */
.lk-topbar-admin { background: linear-gradient(135deg, #9a3412 0%, #ea580c 100%); }

/* and a side-nav variant that follows the platform theme */
.lk-admin-platform .lk-sidenav-item.on { background: #fdebdc; color: #c2410c; }

That’s how the same shell can render in two different palettes — the Java layout adds one extra class to the root, and the cascade does the rest.

Responsive design

The RESPONSIVE section uses media queries to collapse multi-column layouts on narrow screens: 

@media (max-width: 720px) {
  .md-split,
  .prof-split,
  .adm-ann-split { grid-template-columns: 1fr; }

  .ow-edit-split,
  .ow-venue-split { grid-template-columns: 1fr; }
}

Notice the pattern — we don’t rewrite the layout, we just change grid-template-columns from 1.7fr 1fr to a single column.

§ 6What you can actually do

Re-brand in 30 seconds

Change the value of --primary in :root and reload. Every blue surface in the app flips: the topbar, primary buttons, links, side-nav active highlight, badges. You don’t need to touch a single Java file.

Add a token for a feature

Say you want a new “success-soft” color for a celebration banner. Add it once: 

:root {
  /* …existing tokens… */
  --ok-soft: #dcfce7;
}

Now any class can use background: var(--ok-soft);. Twenty rules, one source of truth.

Add a kit modifier

Need a new button variant for a soft “news” accent? Drop a sibling rule: 

.lk-btn-news {
  background: #fff7ed;
  color: #9a3412;
  border-color: #fed7aa;
}
.lk-btn-news:hover { background: #ffedd5; }

Then from Java: btn.addClassName("lk-btn-news");. That’s it.

§ 7Exercises

EX 3Re-brand to purple

Find :root in styles.css. Change three values:

--primary:      #6d28d9;
--primary-light:#7c3aed;
--primary-dark: #5b21b6;

Save the file. The dev server hot-reloads the CSS. Browse to / — everything blue is now purple. Buttons, topbar, links, the side-nav active state. Revert when done.

EX 4Add a glowing primary button on hover

Find the BUTTONS section. Add a single rule:

.lk-btn-primary:hover {
  box-shadow: 0 0 0 4px rgba(26, 84, 144, 0.18);
}

Reload. Hover any “Sign in” or “Save” button. Now try with transition: box-shadow .15s on .lk-btn for a smoother effect.

EX 5Make the admin shell green instead of orange

Find .lk-topbar-admin. Swap the gradient:

.lk-topbar-admin {
  background: linear-gradient(135deg, #15803d 0%, #22c55e 100%);
}
.lk-admin-platform .lk-sidenav-item.on {
  background: #dcfce7;
  color: #15803d;
}
.lk-admin-platform .lk-sidenav-ico {
  color: #15803d;
}

Sign in at /admin/sign-in (admin / admin123). The whole platform-admin workspace is green.

EX 6Force-collapse the owner edit-split at a higher breakpoint

Find @media (max-width: 720px). Bump the breakpoint to 980px so the owner forms stack earlier:

@media (max-width: 980px) {
  .ow-edit-split, .ow-venue-split { grid-template-columns: 1fr; }
}

Resize the browser to ~900px. The Event Management form should now be single-column. Revert when done — this is just to see how the cascade works.

EX 7Add a brand-new component without touching Java

Add a celebratory ribbon class:

.lk-ribbon {
  display: inline-block;
  padding: 4px 14px;
  background: linear-gradient(90deg, var(--accent), #facc15);
  color: #fff;
  font-weight: 800;
  font-size: 11px;
  text-transform: uppercase;
  letter-spacing: .08em;
  border-radius: 999px;
  box-shadow: 0 2px 8px rgba(244, 121, 32, 0.35);
}

Then in any Java view, do Span tag = new Span("NEW"); tag.addClassName("lk-ribbon"); and add it to a card. You just shipped a new visual primitive.

EX 8Tweak theme.json — remove utility

Remove "utility" from lumoImports. Restart the app (theme.json changes need a full restart, not just a hot reload). The kit views work fine because we use our own classes — this proves we’re not relying on Lumo utilities. Put it back when done.

§ 8Things to remember

  • Hot reload is per-file. Edits to styles.css apply within ~1s in dev mode. Edits to theme.json need a server restart.
  • Lumo tokens vs. our tokens. Vaadin’s built-in components consume Lumo tokens (--lumo-*). Our kit components consume our tokens (--primary, --accent, …). Mostly we use our kit, so day-to-day branding lives in :root.
  • One global namespace. The .lk-* / .bz-* / .vk-* prefix is the discipline that keeps it sane.
  • generated/ is build output. Never commit edits there. Delete it freely.
  • Don’t fight the cascade. If something doesn’t take effect, open DevTools > Computed and see which rule is winning. Specificity beats source order; !important beats specificity; sane prefixes beat !important.
