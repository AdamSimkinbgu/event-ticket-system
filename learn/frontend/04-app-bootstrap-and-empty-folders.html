App bootstrap & the empty folders

How a Vaadin Flow + Spring Boot app actually starts up — EventTicketSystemApplication, AppShell, and a frank look at the two folders sitting empty (controllers/, middleware/) and whether we even need them.

§ 1How a Vaadin Flow + Spring Boot app actually starts

You'd expect a "main()" somewhere. There is — it's a one-liner. But what actually happens when you run ./mvnw spring-boot:run is interesting because Spring Boot and Vaadin Flow are two frameworks that have learned to share an HTTP server. Knowing the choreography is the difference between "it just works" and "I can debug when it doesn't."

The 30-second version

  1. Spring Boot starts. Spring scans the classpath, builds the application context (DataSource, JPA, Spring Security if configured, every @Component/@Service/@Repository), and starts the embedded Tomcat on port 8080.
  2. Vaadin Flow registers itself as a Spring component during context init. It hooks into Spring Boot's ServletContainer and registers SpringServlet as the catch-all servlet for the app.
  3. Vaadin scans for AppShellConfigurator. Finds AppShell.java, reads its @Theme("tickethub"), and queues up the theme assets (Lumo modules listed in theme.json + our styles.css).
  4. Vaadin scans for @Route. Every routed view in views/ is registered with a path and a layout.
  5. Spring Security (if configured) registers filter chains in front of the Vaadin servlet — meaning every Vaadin request goes through filters first.
  6. Browser hits localhost:8080/. Tomcat → Spring filter chain → SpringServlet → Vaadin's route resolver finds LandingView → layout chain renders → HTML+JS streams back.

Most of the heavy lifting is invisible. The two files we own are tiny.

§ 2EventTicketSystemApplication — the Spring entry point

src/main/java/com/ticketing/system/EventTicketSystemApplication.java 
package com.ticketing.system;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class EventTicketSystemApplication {

    public static void main(String[] args) {
        SpringApplication.run(EventTicketSystemApplication.class, args);
    }
}

Ten lines. Three things going on:

1. The package matters

com.ticketing.system is the root package — everything else (Presentation, Core, Infrastructure) lives below it. Spring's component scan starts from the package of the class annotated with @SpringBootApplication and walks downwards. Move this class one level down and Spring stops seeing half the codebase.

2. @SpringBootApplication is three annotations in one

Meta-annotationWhat it brings
@SpringBootConfigurationThe class is a Spring @Configuration source.
@EnableAutoConfigurationSpring inspects the classpath. Finds spring-boot-starter-web? Auto-configures Tomcat. Finds spring-boot-starter-data-jpa? Auto-configures Hibernate + the DataSource from application.yml. Finds vaadin-spring-boot-starter? Registers Vaadin's servlet.
@ComponentScanScans this package + sub-packages for @Component, @Service, @Repository, @RestController, @VaadinSessionScope, etc.

3. SpringApplication.run(…) is the actual boot call

It returns a ConfigurableApplicationContext — the live application context — but we don't capture it because the app runs as a server until killed.

The entire file is intentionally trivial. All real configuration lives elsewhere: application.yml (database, JWT, session TTLs), Infrastructure/config/SecurityConfig.java (Spring Security filter chain), Infrastructure/scheduling/SchedulingConfig.java (background jobs). Keep this file boring — if you ever feel the urge to add @Bean definitions here, they belong in a dedicated @Configuration class.

§ 3AppShell — the Vaadin shell configurator

src/main/java/com/ticketing/system/Presentation/AppShell.java 
package com.ticketing.system.Presentation;

import com.vaadin.flow.component.page.AppShellConfigurator;
import com.vaadin.flow.theme.Theme;

@Theme("tickethub")
public class AppShell implements AppShellConfigurator {
}

Even smaller. The whole job:

1. @Theme("tickethub") activates the custom theme

This line tells Vaadin to look in src/main/frontend/themes/tickethub/ for the theme assets — the theme.json (Lumo modules) and styles.css (everything 01-frontend-folder.html covered). Drop the annotation and you get Vaadin's default Lumo without any of our customisations — no kit classes, no gradients, no auth-card centring.

2. implements AppShellConfigurator is Vaadin's "decorate the HTML shell"

Vaadin discovers any class implementing this interface at startup. The interface has many optional methods you can override:

Method to overrideUse case
configurePage(AppShellSettings)Adjust the viewport meta tag, add favicons, inject extra <link>/<script> tags.
configureLoadingIndicator()Customise the spinner Vaadin shows during long server roundtrips.
@Push on the classEnable WebSocket push for real-time updates (V2-F-07 #212 territory, currently Tier C exempt).
@PWA(name, shortName)Make the app installable as a progressive web app.

We override none of them. The empty implementation just exists so the @Theme annotation has somewhere to live (the annotation only takes effect on a class that implements AppShellConfigurator).

EX 1Add a favicon and a loading-indicator tweak

Drop these into AppShell.java:

@Override
public void configurePage(AppShellSettings settings) {
    settings.addFavIcon("icon", "icons/favicon.ico", "32x32");
    settings.addMetaTag("theme-color", "#1a5490");  // match the kit's --primary
}

@Override
public void configureLoadingIndicator() {
    // Don't show the spinner until 700ms after a click — most calls finish faster.
    LoadingIndicatorConfiguration cfg = LoadingIndicatorConfiguration.default();
    cfg.setFirstDelay(700);
    cfg.setSecondDelay(2000);
    cfg.setThirdDelay(5000);
}

Then put a 32×32 PNG/ICO at src/main/frontend/icons/favicon.ico. Reload — the browser tab now shows a proper icon and the loading bar feels snappier.

§ 5Presentation/controllers/ empty

src/main/java/com/ticketing/system/Presentation/controllers/

Right now this folder contains exactly zero files. The reason it exists at all is forward-looking scaffolding: when V2-F-04 (#210) ships REST controllers (ApiResponse<T> wrapper, JSON endpoints for external integrations), they'll land here.

What it would contain

Spring @RestController classes that expose HTTP endpoints. Example shape, when the folder eventually fills: 

@RestController
@RequestMapping("/api/v1/events")
public class EventCatalogController {

    private final EventQueryService events;

    @GetMapping
    public ApiResponse<Page<EventCardDTO>> list(@RequestParam(required = false) String category) {
        return ApiResponse.ok(events.list(category));
    }

    @GetMapping("/{id}")
    public ApiResponse<EventDetailsDTO> one(@PathVariable long id) {
        return ApiResponse.ok(events.findById(id));
    }
}

Why presentation/controllers and not somewhere else?

Because REST controllers are a presentation concern — they translate HTTP shape into domain calls, just like Vaadin views translate clicks into domain calls. Both are different flavors of "user interface". A view is a graphical interface; a REST controller is a machine interface for the same operations.

The alternative layout puts controllers under Infrastructure/ or Web/ instead. Both work. We went with Presentation/controllers/ because the architectural intent is: "anything user-facing, graphical or not, lives in Presentation/".

§ 6Presentation/middleware/ empty

src/main/java/com/ticketing/system/Presentation/middleware/

Same situation, different concept. Right now: empty. When it fills, it'll hold Spring filters or HTTP interceptors that wrap requests/responses with cross-cutting behavior.

What it might contain (when V2 wiring lands)

  • A request-ID filter — tag every request with a UUID, propagate it into logs (SLR.8.1 #158 territory).
  • A CSRF token check for the REST controllers (Vaadin Flow handles its own CSRF, but REST endpoints need their own filter).
  • An audit-log filter for admin endpoints — record who called what (V2-LOG-02 #236 / SLR.2 #148).
  • A structured-error mapper that converts uncaught exceptions into the ErrorPayload shape (V2-ERROR-01 #255).

Spring already wires the Filter chain for us — we just drop classes annotated @Component implementing jakarta.servlet.Filter or extending OncePerRequestFilter into this folder and Spring picks them up.

"Middleware" is loose vocabulary. In Java/Spring the precise terms are filter (servlet-level, wraps every request) and interceptor (Spring MVC-level, wraps controller invocations only). "Middleware" is closer to the Node.js / Go vocabulary. We named the folder generically because the V2 plan didn't pre-decide which Spring abstraction to use — the choice depends on which kind of cross-cutting concern lands first.

§ 7Do we even need them?

A reasonable question, given they've been empty since the project started. The honest answer comes in two parts.

Right now — no

The V2 placeholder UI is pure Vaadin Flow. Every user interaction is a Java method call. There's no JSON, no AJAX endpoint, no public REST API. So:

  • No REST controllers needed — Vaadin handles all the HTTP routing internally.
  • No middleware needed — Vaadin handles CSRF, session, and serialisation; AuthBootstrap handles auth/RBAC gating inside the Vaadin servlet.

Eventually — yes, but only some of it

Use caseWill fill controllers/?Will fill middleware/?
Mobile app talking to our backendYes — REST endpoints for the catalog, cart, checkout.Yes — JWT auth filter, CSRF, rate limit.
Webhook from payment providerYes — one @PostMapping controller.Maybe — signature verification filter.
Server-Sent Events / streaming notificationsYes — SSE endpoint.No.
Just a public health checkOne trivial controller — or just use Spring Actuator's built-in /actuator/health.No.
Adding audit logging for admin actionsNo.Yes — a filter or HandlerInterceptor.
Adding a global structured-error mapperNo — @ControllerAdvice instead.Yes — if it spans Vaadin too.

Why we keep the folders even though they're empty

Two reasons:

  1. The architectural intent is documented by the file tree itself. A new contributor opening Presentation/ immediately sees that REST controllers and middleware have a home reserved. No need to read a doc to learn that.
  2. Empty folders survive the cleanup pass. If we deleted them now, someone re-adding REST controllers six months from now would have to decide where to put them — and the decision was already made.
Empty folders aren't free. Git doesn't track empty directories. For these folders to survive a fresh clone, there has to be at least one file in each — typically .gitkeep. If you ever see controllers/ missing after cloning, it's because .gitkeep got dropped. Add a fresh one and re-commit.

EX 2Add the first real controller as a smoke test

Drop this in controllers/HelloController.java: 

@RestController
public class HelloController {

    @GetMapping("/api/hello")
    public Map<String, Object> hello() {
        return Map.of(
            "app", "event-ticket-system",
            "version", "V2-placeholder",
            "now", Instant.now().toString()
        );
    }
}

Run the app, hit curl localhost:8080/api/hello. You should see JSON back — proving the controller folder is reachable. Delete the file when you're done; we don't want the placeholder endpoint shipping. (When V2-F-04 #210 lands, the actual REST scaffolding will replace this one-liner.)

§ 8Things to remember

  • Two bootstrap files, both trivial. EventTicketSystemApplication is the Spring entry; AppShell is the Vaadin theme/shell configurator. Don't inflate them — config belongs in dedicated @Configuration classes.
  • Spring + Vaadin share Tomcat. Spring boots Tomcat, Vaadin registers itself as the catch-all servlet, Spring Security filters wrap everything. When auth eventually goes real, that's where it inserts itself — not in AuthBootstrap, which becomes a Vaadin-internal complement to Spring Security.
  • The two empty folders are placeholders for V2-F-04 REST controllers and future Spring filters/interceptors. Empty today, intentionally reserved space.
  • We don't need them right now because the V2 placeholder UI is pure Vaadin Flow. We'll need controllers when a mobile app, webhook, or SSE endpoint shows up. We'll need middleware when audit logging, structured errors, or REST-side JWT enters the picture.
  • "Middleware" is loose vocabulary in Java. Use filter (servlet-level) or interceptor (Spring MVC) depending on where the concern lives. The folder name is generic on purpose.
  • @Theme("tickethub") is the load-bearing line in AppShell. Drop it and your custom CSS stops shipping.
  • .gitkeep files are what keep the empty folders alive in git. If a folder disappears after cloning, this is why.
