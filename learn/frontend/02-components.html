Presentation/components

A complete tour of the Vaadin component kit — 44 files across 4 sub-packages, what they are, how they compose, and the design tricks that make them work.

§ 1Orientation

The components/ tree is the Vaadin equivalent of a React design system — every Java class here renders a piece of DOM with kit CSS classes attached. Views compose these components into screens; layouts compose them into shells. By the end of this page you should know which class to reach for in any situation, how to extend the kit, and where each piece is wired up.

Folder layout

Presentation/components/
├── Toasts.java                // 72 LOC — the only file at the root
├── kit/                       // 30 LkXxx primitives — the shared design system
├── venue/                     // 7 files — seat picker + policy editor
├── messaging/                 //  3 files — master/detail inbox parts
└── buyer/                     //  3 files — tickets + posters + refund modal

Design patterns you'll see everywhere

  • Subclass an HTML element. Every component extends a concrete Vaadin HTML class (Div, Span, NativeButton, NativeLabel, Section, Header, Aside, Footer). The DOM tag is part of the type.
  • Fluent setters. Configuration methods return this so calls chain: new LkBtn("Save").variant(primary).size(l).full();.
  • CSS-only theming. Java attaches the class names; styles.css owns all visual rules. Java never sets fonts, colors, padding (with rare exceptions for one-off geometry).
  • Innermost slot insertion. Several components override add() so children go to a body slot instead of the root (LkCard, LkAuthCard, LkPage, PolicyGroup). That keeps the title/header above content automatically.
  • Tiny record data classes. Where a component needs a structured payload it declares a Java record right inside the class (LkSideNav.Item, LkTopBar.NavItem, VkVenueMap.Zone, MdThread.Message, BzTicketDialog.TicketInfo). Immutable, no boilerplate.
  • Static factories for variants. When a class can render in obviously distinct flavors, it exposes public static factory methods. See LkAccountMenu.buyer(…), LkNotifPanel.admin(), LkSearchPanel.defaults().
Mental model. Each Java class is roughly: "a few CSS classes worth of DOM, plus a fluent API for plugging children in". Once you internalize that, the whole kit fits in your head.

§ 2rootToasts

Toasts — 72 LOC · utility class (no instances)

The single file at components/Toasts.java wraps Vaadin's com.vaadin.flow.component.notification.Notification into three top-of-screen popup helpers that the whole codebase uses for action feedback. It exists because calling Notification.show("Done") directly from a view is an explicit anti-pattern listed in docs/vaadin-standards.html — we want one place where every success / failure / warn looks the same and carries a dismiss "×" button.

API

public static void success(String msg);   // green theme, 4s
public static void failure(String msg);   // red theme, 6s — for error paths
public static void warn(String msg);      // amber theme, 5s — near-expiry / "be careful"

How it's built

The private toast(msg, variant, duration) method builds a flex Div with the message Span + an inline tertiary Button whose icon is an LkIcon of name "close" at size 14. The close button calls n.close() so users can dismiss without waiting for auto-close. Notifications are anchored at Position.TOP_END and pushed below the 56px topbar by the kit's .lk-topbar + toast-positioning CSS so they never overlap the bell / avatar.

Usage

// inside any submit handler
if (username.isEmpty()) {
    Toasts.failure("Please enter a username.");
    return;
}
MockAuth.signIn(username.getValue());
Toasts.success("Signed in as " + username.getValue());
Why a final utility class with private constructor? No state, no instances ever needed. The private constructor + final on the class is Java's idiomatic way of saying "this is a namespace for static methods, don't even think about new Toasts()".

§ 3kitFoundation primitives

These are the building blocks every other kit component uses. Six files: Lk (text / heading factories), LkPage (page envelope), LkCol / LkRow (flex primitives), plus LkChip and LkStatusDot for tiny status atoms.

Lk — 62 LOC · static factory class

A pure utility namespace mirroring React's tiny H1, H2, H3, Txt helpers. Each method returns a freshly-built component pre-decorated with the right kit class.

API

MethodReturnsClass attached
h1(text)H1.lk-h1
h2(text)H2.lk-h2
h3(text)H3.lk-h3
txt(text)Span—
muted(text)Spaninline color:var(--muted)
mono(text)Span.lk-mono
faint(text)Spaninline color:var(--faint)
divider()Div.bz-divider

Why this exists instead of just calling new H2("foo") + addClassName? Because most sections need a kit-styled heading, and forgetting the class name is the single most common inconsistency bug. Lk.h2("Past announcements") can't forget the class.

LkPage — 80 LOC · extends Div

Every routed view extends LkPage. It owns the .lk-page surface and provides three top-of-page slots: the H1 title, an optional subtitle, and a right-aligned actions strip. Anything else you add(…) goes into the body, stacked with the kit's vertical-gap rhythm.

Internal DOM

<div class="lk-page">
  <div class="lk-page-head">
    <div>            // titleCol
      <h1 class="lk-h1">{title}</h1>
      <div class="lk-page-sub">{subtitle?}</div>
    </div>
    <div class="lk-page-actions">{actions?}</div>
  </div>
  {body children — added later}
</div>

API

new LkPage()                          // empty
new LkPage("My events")              // title set
.title("My events")
.subtitle("Owned by Live Nation Israel.")
.actions(new LkBtn("New event").variant(primary))
.add(filterRow, gridCard);              // body

Implementation detail — ensureHead()

The head row is held in a field but not attached to the DOM at construction. Only when you call title, subtitle, or actions does ensureHead() insert it as the first child via getElement().insertChild(0, headRow.getElement()). That's why empty pages don't get a stray header block, and the head appears above any body children regardless of call order.

EX 1Add a "back" button to LkPage

What if you wanted a back arrow in front of every title? Add a method:

public LkPage backLink(Class<? extends Component> target) {
    var link = new RouterLink("← Back", target);
    link.addClassName("lk-page-back");
    titleCol.getElement().insertChild(0, link.getElement());
    return this;
}

Add .lk-page-back{display:inline-block;color:var(--muted);font-size:13px;margin-bottom:6px} to styles.css. Every page can now do .backLink(BrowseEventsView.class).

LkCol & LkRow — 25 + 40 LOC · extend Div

The flex primitives. LkCol is a vertical column, LkRow is a horizontal row with wrap enabled by default. Both default to a 12px gap. 

new LkCol(child1, child2, child3)         // flex-direction:column, gap:12px
new LkRow(child1, child2).gap(8).noWrap()  // flex-direction:row, gap:8, no wrap
                                          // .justify("flex-end") moves children right
                                          // .align("flex-start") top-aligns

Use these instead of HorizontalLayout / VerticalLayout — the Vaadin ones embed their own theming, whereas ours render as plain divs that obey our CSS rhythm.

LkChip — 38 LOC · extends Span

A pill-shaped chip used for category filter buttons, "recent search" entries, and as the base visual for LkFilterChip. Optional caret ▾ at the end via withCaret(); toggle the .on state with active(boolean). 

new LkChip("Concerts").active()           // highlighted chip
new LkChip(new LkIcon("music", 13), "Concerts")  // icon + label
new LkChip("Status").withCaret()           // dropdown trigger look

LkStatusDot — 24 LOC · extends Span

Coloured filled circle + label, used in tables and detail rows to indicate state. Four tones (ok, warn, muted, err) drive the dot color via .lk-dot-{tone}. 

new LkStatusDot(Tone.ok,   "Active")
new LkStatusDot(Tone.warn, "Pending")
new LkStatusDot(Tone.err,  "Cancelled")

§ 4kitButtons & icons

LkBtn — 72 LOC · extends NativeButton

The kit's button. Renders an actual <button> (not a Vaadin vaadin-button) so it's lightweight and styles purely from our CSS. Inner Span holds the label so the icon can sit beside it.

Variants & sizes

VariantCSS classUse for
primary.lk-btn-primaryThe single most-important action on a screen
secondary.lk-btn-secondaryDefault. White card with border
tertiary.lk-btn-tertiaryBare-link style (Cancel, Discard)
error.lk-btn-errorDanger zone (Cancel event, Revoke)
success.lk-btn-successConfirmation actions
GHOST_LIGHT (constant).lk-btn-ghost-lightTranslucent button used on dark topbar (e.g. Register)

Sizes s / m / l map to .lk-btn-{s,m,l}; default is m.

Example

new LkBtn("Save policy")
    .variant(LkBtn.Variant.primary)
    .icon(new LkIcon("check", 16))
    .size(LkBtn.Size.l)
    .full()
    .onClick(e -> presenter.save());

Implementation note

When you call variant() twice in a row, the first call's class isn't left behind — the method loops over the enum and removeClassName(val.cls()) before adding the new one. Same in size(). That's why repeatedly toggling variants doesn't leak stale classes.

LkIconBtn — 20 LOC · extends NativeButton

A 32×32 square button for icon-only actions — row icons in grids, the close on a toast, the menu trigger. Just attaches .lk-iconbtn and wraps the icon. Optional constructor sets title attribute for the tooltip. 

new LkIconBtn(new LkIcon("trash", 15), "Revoke")
    .addClickListener(e -> confirmRevoke());

LkIcon — 130 LOC · extends Span

This is the longest and most interesting "primitive" in the kit. It renders an inline SVG glyph — one of 43 named shapes — with arbitrary size and stroke-width.

How it draws

The class holds a static final Map<String, String[]> called PATHS that maps glyph names ("ticket", "building", "comment", …) to arrays of SVG path-d strings. The constructor calls buildSvg(name, size, stroke) which assembles an SVG string and sets it as innerHTML on the wrapping <span class="lk-icon">. 

String svg = "<svg xmlns='http://www.w3.org/2000/svg' width='" + size +
            "' height='" + size + "' viewBox='0 0 24 24' fill='none' " +
            "stroke='currentColor' stroke-width='" + stroke +
            "' stroke-linecap='round' stroke-linejoin='round' aria-hidden='true'>";
for (String p : paths) svg += "<path d='" + p + "'/>";
svg += "</svg>";
getElement().setProperty("innerHTML", svg);

Why innerHTML and not Vaadin's Html component?

The javadoc explains it explicitly: Vaadin's Html goes through Jsoup, which is HTML5-aware but loses the SVG namespace and silently drops the paths. Setting innerHTML on the element directly lets the browser's HTML5 parser handle the inline SVG correctly. This is the kind of thing you wouldn't guess from first principles — that sentence in the javadoc represents an afternoon of debugging.

Why stroke="currentColor"?

Because that makes the icon inherit the surrounding text color. An icon inside a primary blue button is white. The same icon inside a muted help text is grey. No need to recolor — the SVG just follows CSS.

Why size + stroke are configurable

You'll see new LkIcon("check", 11, 2.6) inside LkCheckRow. At 11px the default 1.7px stroke would look spindly, so a thicker stroke keeps the check visible. Most callers use the default 1.7.

Available glyphs

The 43-glyph set covers nav (menu, caret, arrowLeft, arrowRight), domain (ticket, seat, cart, building, policy, chart, org, crown, briefcase), state (check, close, plus, minus, warning, checkCircle, info, clock, lock, star, trophy, crosshair), and decoration (music, mic, theater, map, flask, users, card, bell, calendar, mobile, edit, eye, trash, copy, more, search, gear, logout, comment).

Unknown name? PATHS.getOrDefault(name, new String[0]) returns an empty array — the resulting SVG has no <path> children and renders as blank space. You won't get a runtime error, you'll get a missing-icon mystery. This is exactly what bit the Send announcement button when it asked for a non-existent "send" icon — the bug only surfaced visually.

EX 2Add a new icon

Suppose you want a "heart" glyph for favorites. Find a 24×24 SVG path and add it to the static initializer:

PATHS.put("heart", new String[] {
    "M12 21s-7-4.35-9.5-9A5.5 5.5 0 0 1 12 6a5.5 5.5 0 0 1 9.5 6c-2.5 4.65-9.5 9-9.5 9Z"
});

Now new LkIcon("heart", 18) works everywhere. The whole rest of the codebase consumes it via the same constructor.

LkBadge — 42 LOC · extends Span

Pill-shaped status label sitting next to titles and inside table cells. Seven tones: muted, success, warn, warning, error, primary, contrast. warn and warning are aliases — both exist because the React reference used warning and the Vaadin layer settled on warn; keeping both lets either naming compile. 

new LkBadge("Active", Tone.success)
new LkBadge("Cancelled", Tone.error).small()  // adds .sm modifier
new LkBadge("LIVE", Tone.success).small()

The pattern with tone() mirrors LkBtn.variant() — loop over every value, removeClassName, then add the new one. Toggling tones leaves no stale classes.

§ 5kitSurface containers

"Surface" components are the ones that hold content: LkCard for everything boxed, LkBanner for info/warn strips, LkTile for clickable navigation cards, LkStat for KPI displays, LkAuthCard for sign-in/register pages.

LkCard — 121 LOC · extends Section

The most-used surface in the entire app. Sits on every page that shows grouped content. Four optional pieces — title, subtitle, header-right slot, footer — plus a body that holds everything else.

Internal DOM

<section class="lk-card">
  <header class="lk-card-h">     // only if title/subtitle/headerRight was set
    <div>
      <div class="lk-card-title">{title}</div>
      <div class="lk-card-sub">{subtitle?}</div>
    </div>
    <div>{headerRight?}</div>
  </header>
  <div class="lk-card-body" style="padding:20px">
    {body children}
  </div>
  <footer class="lk-card-f">        // only if footer() was called
    {footer children}
  </footer>
</section>

API

MethodEffect
title(s) / subtitle(s)Set/replace text. Empty string removes.
headerRight(c)Right-aligned slot inside the header (e.g. a "View all" link).
pad(px)Override body padding. .pad(0) when the body is a borderless grid.
accent()Adds .lk-card-accent — blue left border.
danger()Adds .lk-card-danger — red border (Cancel event zone).
flush()Removes border-radius and outer margin — for cards inside cards.
footer(…)Creates / replaces the bottom <footer> region.

The clever bit — overridden add()

When you call card.add(myChart, myDescription), you expect those to land in the body — not as siblings of the header. But our own constructor also does add(body) internally, which would recurse forever if add always redirected to body. The override checks: if the child is the card's own header / body / footer reference, fall through to super.add; otherwise redirect to body: 

public void add(Component... components) {
    for (Component c : components) {
        if (c == header || c == body || c == footer)
            super.add(c);
        else
            body.add(c);
    }
}

This is the pattern used by LkAuthCard and LkPage too.

Lazy header

The header is created in the constructor but not attached. Only when one of title, subtitle, headerRight is called does ensureHeader() insert it at index 0: 

private void ensureHeader() {
    if (headerActive) return;
    getElement().insertChild(0, header.getElement());
    headerActive = true;
}

That keeps an empty card from showing a stray header bar.

LkBanner — 71 LOC · extends Div

Coloured strip with leading icon + message + optional trailing action slot. Two tones: warn (amber) and info (blue). Used for "Cycle prevention is enforced", "Validation rejects empty groups", "Refunds are processed in 3–5 days", etc. 

new LkBanner(LkBanner.Tone.warn, new LkIcon("warning", 17),
    "You cannot appoint someone who already appointed you.");

The three slots (icon, body, action) are pre-created Span/Div fields. You can later replace any of them via setIcon, setBody(Component...), or setAction. The body slot has flex:1 min-width:0 so it absorbs all spare horizontal space and truncates gracefully.

LkTile — 32 LOC · extends NativeButton

Big clickable card for dashboard navigation — OwnerDashboardView and AdminDashboardView show grids of these. Renders as a button (not a Div with click handler) so the browser's native focus ring and keyboard activation work for free.

Anatomy

<button class="lk-tile">
  <span class="lk-tile-icon">{icon}</span>
  <span class="lk-tile-title">{title}</span>
  <span class="lk-tile-desc">{description}</span>
  <span class="lk-tile-go">Open →</span>
</button>

"Open →" is hardcoded — intentional consistency. Call active() to add .on for a highlighted "current" tile. Click handlers live on the caller via addClickListener.

LkStat — 40 LOC · extends Div

The KPI card. Title above a big value, optional "delta" line at the bottom with a tone (up, warn, or none). 

new LkStat("Total revenue · 30d", "$1.92M")
    .delta("▲ 8%", LkStat.Tone.up);
Pitfall the smoke test caught. The enum only has up / warn / none — no down. Calling LkStat.Tone.down is a compile error. For negative deltas, use Tone.warn (amber) — that's the deliberate signal that something needs attention. Don't invent a red tone unless the design says so.

LkAuthCard — 91 LOC · extends Div

The centered authentication-page shell used by LoginView and RegisterView. Root is the .auth-wrap flex centering container; the inner .auth-card holds the brand line ("TicketHub" with a ticket icon), an H2 title, a subtitle paragraph, and then whatever you add(…) afterwards.

Convenience helpers

  • col(int gap, Component... cs) — group several components in a flex column with the given gap. Used to bundle username + password + signIn.
  • divider(label) — the "or continue with" horizontal rule with text in the middle.
  • foot(text, linkLabel, linkTarget) — bottom paragraph like "Don't have an account? Create one →" with a real RouterLink.

The pattern is the same as LkCard — overridden add sends children into the inner card div so you never accidentally splice them into the outer wrap.

AdminLoginView doesn't use LkAuthCard because it needs different branding ("Event Ticket Platform · Admin") and an orange severity border — instead it copies the .auth-wrap / .auth-card structure directly. That's a deliberate trade-off: a parameterised brand line would add complexity for one caller.

§ 6kitForm inputs

Seven form components: LkField (mock labeled input), LkCheckRow, LkRadio, LkSelect, LkStepper, LkDateRangeField, LkFilterChip. Note: the kit's "form inputs" are mostly visual mock components — for screens that actually submit data, the views use real Vaadin TextField, EmailField, PasswordField, TextArea, IntegerField, which Lumo styles for free.

When do you use a kit input vs. a real Vaadin field? Kit inputs (LkField, LkRadio, LkStepper) are for placeholder visual fidelity — when you need a form to look right in screenshots but nobody's typing into it yet. Real Vaadin fields (TextField, EmailField, PasswordField, TextArea, IntegerField, EmailField) go on screens that submit. Compare LoginView (real fields) vs. an early placeholder view (kit fields). The kit inputs even fall back gracefully — reading getValue() returns the placeholder text.

LkField — 123 LOC · extends NativeLabel

The mock labeled input. Renders as a <label class="lk-field"> with five inner slots: label, required mark, prefix affix, value, suffix affix — plus an optional helper line below. 

LkField price = new LkField()
    .label("Ticket price").required()
    .prefix("$")
    .value("150")
    .suffix(new LkIcon("info", 14))
    .helper("Includes taxes.");

price.invalid().helper("Price must be positive.");  // switches helper to red

Lazy-attached slots again — prefix/suffix are constructed in the constructor but only inserted into the DOM when prefix() / suffix() is called. This keeps the default appearance clean: no leftover empty spans.

LkCheckRow — 63 LOC · extends Span

A checkbox + label row that toggles on click. The "checkbox" is a styled <span class="lk-checkbox"> which contains a tiny check icon when on. 

new LkCheckRow("Remember me", false);
new LkCheckRow("Manage events", true)
    .onToggle(checked -> presenter.permissionChanged(checked));

Why wrapperClass?

By default the outer span has class .lk-checkrow. The auth screens want their "Remember me" row in a different visual lane (smaller font, no left-alignment with form labels), so LoginView calls .wrapperClass("auth-remember") to swap the class. The inner .lk-checkbox stays the same — only the outer wrapping class changes.

LkRadio — 51 LOC · extends NativeLabel

A labeled group of mutually-exclusive options. Visual only — the on state is set per-option by matching to selected. No two-way binding; if you need real behavior, use RadioButtonGroup from Vaadin. 

new LkRadio()
    .label("Zone type")
    .options("Seated", "Seated", "Standing");
//      selected ^      ^^^^ options in order

LkSelect — 101 LOC · extends NativeLabel

A real working select. The trigger looks like an LkField but clicking it opens an LkMenu via LkPopover. Each option click updates the value, fires onChange, and rebuilds the menu so the new selection shows the .on tick. 

LkSelect category = new LkSelect("All", List.of("All", "Music", "Sport", "Theatre"))
    .label("Category")
    .onChange(val -> presenter.filter(val));

category.getValue();   // returns String (NOT Optional)
category.setValue("Music");  // programmatic set, fires onChange
API trap. The first ctor arg is the initial value, not the label. new LkSelect("Audience", …) would show "Audience" inside the trigger as if it were a chosen option. To set the label, chain .label("Audience"). Also getValue() returns String directly — not an Optional. The smoke test caught both mistakes when AdminAnnouncementsView was first written.

LkStepper — 44 LOC · extends NativeLabel

A "− / value / +" quantity control with optional label. Visual only; the buttons aren't wired to any handler. Used in VenueMapEditorView for the rows/cols stepper, and in VkStandingZone for the quantity selector. 

new LkStepper("6").label("Rows").width("120px");

LkDateRangeField — 107 LOC · extends NativeLabel

A select-like trigger that opens a popover with seven date-range presets ("Any time", "Today", "This weekend", "Next 7 days", "Next 30 days", "This month", "Next 3 months") plus a "Custom range" divider with two date-input stubs. Like LkSelect — the value is real, the custom range UI is mock. 

new LkDateRangeField("Next 30 days")
    .label("When")
    .onChange(val -> presenter.dateFilter(val));

Look at the inner rebuild() — it tears down and rebuilds the entire menu on every selection change. That's simpler than diff-patching, and the menu is small enough (7 items + custom row) that the cost is invisible.

LkFilterChip — 105 LOC · extends Div

The most behavior-rich kit input. A chip labeled like a category ("Status", "Date range", "Company") that opens a popover with check-rows + a "Clear" button + an "Apply" button. Single- or multi-select. 

new LkFilterChip("Status",
    List.of("Open", "Responded", "Resolved", "All"),
    false,                          // multi-select
    List.of("Open"))                  // pre-applied

Chip text reflects state

When the user selects anything, the chip's label gets a count suffix: "Status · 2". When the selection is empty, the suffix is removed. The .on class is toggled on the chip to highlight it.

Single vs. multi

With single=true, the checkboxes get an extra .rnd class so they render as rounded radio circles. Clicking an option replaces the selection rather than adding to it. The "Clear" link appears only when something is selected.

§ 7kitShell & navigation

LkTopBar — 260 LOC · extends Header

The single most important kit class. Every layout in the app composes one. It uses a slot-based fluent API: call methods in order, and they attach DOM in that order. The "right slot" is a single Span.lk-topbar-right that is auto-margined to the right edge by CSS, so anything you add there (bell, cart, account, guest actions) flows from the right.

Three variants

VariantAdds classUsed byVisual
MAIN—MainLayoutBlue topbar (buyer)
ADMIN—WorkspaceLayoutBlue topbar + " › Workspace" crumb
PLATFORM.lk-topbar-adminPlatformAdminLayoutOrange gradient topbar

Compose calls in order

new LkTopBar(Variant.MAIN)
    .brand("TicketHub", LandingView.class)
    .nav(List.of(
        new LkTopBar.NavItem("Browse",     BrowseEventsView.class),
        new LkTopBar.NavItem("My Tickets", MyAccountView.class)
    ), activeLabel)
    .searchDefault()
    .cart(CartView.class, MockCart.size(), shortestDeadlineMs())
    .bellDefault(false)
    .account(initials(name), name, buildMemberMenu(name));

The right slot trick

ensureRightSlot() attaches the right-slot Span the first time any right-side method is called. That way, if you only call brand() + nav() (no bell, no account), the right slot doesn't render at all and there's no orphan flex item pushing nav around.

brand overloads — 4 of them

This is a textbook overload chain. The four overloads progressively expose more knobs: 

brand("TicketHub")                                // just text
brand("TicketHub", " › Workspace")              // + crumb
brand("TicketHub", LandingView.class)            // text + click target
brand("TicketHub", " › Workspace", LandingView.class) // all three

The first three delegate to the fourth. This is the Java pattern for "optional parameters" — Java doesn't have them as a language feature, but overloaded methods give callers the same ergonomic.

The cart(target, itemCount, shortestDeadlineMs) method

The most recently added topbar slot. Three behaviors fused into one call:

  • The cart icon button (always rendered).
  • A numeric badge bubble (only when itemCount > 0).
  • An adjacent live-ticking timer chip (only when shortestDeadlineMs is a future timestamp).

The timer ticks down on the client — one Element.executeJs call sets up a setTimeout chain that updates textContent every second, adds an urgent class when < 3 minutes remain, and self-removes the chip when it reaches 0. The server never has to push a single tick.

Gotcha. Vaadin's Element.executeJs(String, Serializable...) only accepts String, Boolean, Integer, Double, Element, JsonValue. Passing a Long throws IllegalArgumentException. Hence the double endMs = (double) shortestDeadlineMs.longValue() conversion.

The guestActions alternative to account

For logged-out users, instead of an avatar+menu, the topbar shows "Sign in" + a "Register" ghost button. Calling either account() or guestActions() on the same topbar instance is fine — they both target the right slot.

LkSideNav — 102 LOC · extends Aside

The left drawer in the owner workspace and platform-admin shells. A heading + a list of icon+label items. Each item has a RouterLink wrapping the icon & label so the whole row navigates without JavaScript.

The data shape

public record Item(String iconName, String label,
                   Class<? extends Component> target) {}

Active highlight without rebuild

The expensive operation is items(List, activeLabel) — it tears down every existing row and rebuilds them. To highlight the current view without paying that cost, layouts call setActive(label) which only toggles the .on class: 

public LkSideNav setActive(String label) {
    for (int i = 0; i < itemEls.size(); i++) {
        if (itemDefs.get(i).label().equals(label))
            itemEls.get(i).addClassName("on");
        else
            itemEls.get(i).removeClassName("on");
    }
    return this;
}

That's called from afterNavigation in WorkspaceLayout and PlatformAdminLayout on every navigation event.

Why the platform() modifier

Adding .lk-admin-platform to the side-nav root makes the kit's .lk-admin-platform .lk-sidenav-item.on selector fire, painting the active highlight orange instead of blue. The orange-theme owner workspace and the platform admin both benefit.

EX 3Add CSV export

The kit has no built-in export. Add a method:

public String toCsv() {
    var sb = new StringBuilder();
    for (Col c : columns) sb.append(c.header).append(',');
    sb.setCharAt(sb.length() - 1, '\n');
    for (Object r : rows) {
        @SuppressWarnings("unchecked")
        var row = (Map<String, Object>) r;
        for (Col c : columns) {
            var cell = row.get(c.key);
            sb.append(cell == null ? "" : cell.toString().replace(",", ";")).append(',');
        }
        sb.setCharAt(sb.length() - 1, '\n');
    }
    return sb.toString();
}

Then wire your "Export CSV" button to UI.getCurrent().getPage().setLocation(...) with a data-URI. Components shipped to the cell as Component would need their getText() equivalent — or you keep an internal copy of the string value.

§ 11venuePolicy editor

The policy editor screen (PurchasePolicyEditorView) was originally rendered with two kit components, PolicyGroup and PolicyRule — left-bracket card stacks composed recursively. That design didn't hold up when we needed proper pan/zoom, drag, and connector geometry that stayed tidy as the tree grew. The live editor now uses PolicyTreeMap, a Vaadin wrapper around a hand-written D3.js SVG renderer.

PolicyTreeMap — ~70 LOC Java + ~150 LOC JS · extends Div

Lives in its own sub-package (components/policy/) because it's the only kit component with a JS-side counterpart. The Java half is a thin wrapper; the JS half does the work.

How it's wired

@JsModule("./policy-tree-map.js")
public class PolicyTreeMap extends Div {

    private BiConsumer<String, String> onAction;
    private String pendingJson;

    public PolicyTreeMap() {
        addClassName("policy-tree-map");
        getStyle().set("height", "560px") // … dotted-background, border, etc.;

        addAttachListener(e -> { if (pendingJson != null) callRender(pendingJson); });
    }

    public void render(String treeJson) {
        if (isAttached()) callRender(treeJson);
        else pendingJson = treeJson;
    }

    private void callRender(String treeJson) {
        getElement().executeJs("window.renderPolicyTree(this, $0);", treeJson);
    }

    @ClientCallable
    private void onNodeAction(String nodeId, String action) {
        if (onAction != null) onAction.accept(nodeId, action);
    }
}

The three integration points to remember

  • @JsModule("./policy-tree-map.js") — relative to src/main/frontend/. Vaadin's Vite picks it up automatically, no package.json changes needed.
  • Element.executeJs(…) — pushes data to the client. The JS module's side-effect-on-load registers window.renderPolicyTree, so the executeJs call is one line.
  • @ClientCallable — receives events back from the JS. The JS side dispatches container.$server.onNodeAction(id, action) — $server is the proxy Vaadin attaches to every element that has a @ClientCallable method.

The pending-JSON pattern

executeJs queues until attach, but our renderPolicyTree needs $server set before it runs (or click callbacks become no-ops). So PolicyTreeMap buffers the first render(json) call and replays it from the attach listener. After that, every subsequent render goes straight through.

The JS side — src/main/frontend/policy-tree-map.js

Loads D3 dynamically from a CDN (cdn.jsdelivr.net/npm/d3@7/+esm) on first render. The /* @vite-ignore */ hint stops Vite from trying to resolve the URL at bundle time — the browser resolves it at runtime instead. 

const D3_URL = 'https://cdn.jsdelivr.net/npm/d3@7/+esm';
let d3Promise = null;
function getD3() {
    if (!d3Promise) d3Promise = import(/* @vite-ignore */ D3_URL);
    return d3Promise;
}

window.renderPolicyTree = async function(container, treeJson) {
    const d3 = await getD3();
    const data = JSON.parse(treeJson);
    // d3.tree() lays out positions, d3.zoom() handles pan/zoom,
    // d3.linkVertical() draws curved Bezier connectors…
};

Why pre-built D3 from CDN, not npm

We could install d3 via npm and let Vite bundle it. We don't because:

  1. No package.json diff. Vaadin owns that file; manual edits get clobbered.
  2. One JS module ≪ ~50KB. CDN cache hit beats bundle bloat.
  3. If we ever go offline-first, drop D3 in src/main/frontend/d3.min.js and change one import URL. No rebuild plumbing.

The JSON the Java side ships

{
  "id": "n-abc12345",
  "kind": "comp",
  "op": "AND",
  "label": "AND · all of these",
  "children": [
    { "id": "n-def67890", "kind": "rule", "label": "Age at least 18 years" },
    { "id": "n-ghi13579", "kind": "comp", "op": "OR", "label": "OR · any of these",
      "children": [ /* … */ ] }
  ]

Each node carries a UUID-fragment id that the Java side puts in a Map<String, Node>. When the JS reports onNodeAction(id, action), the Java handler looks the id up and mutates the model accordingly. Then render(treeToJson(root)) is called again and the SVG rebuilds from scratch — the model is the source of truth.

Case 3 — The policy editor tree (Java ↔ D3.js)

PurchasePolicyEditorView is the only screen with a JS-side counterpart. The Java side owns the model (a sealed Node hierarchy of Composite and Rule) and serialises it to JSON; the JS side renders SVG via D3 and reports back when the user interacts. 

// Java side — owns the model, dispatches actions
private PolicyTreeMap mapPanel;
private final Map<String, Node> nodeMap = new HashMap<>();

public PurchasePolicyEditorView() {
    mapPanel = new PolicyTreeMap();
    mapPanel.setOnAction(this::handleNodeAction);   // (id, action) → mutate model
    add(mapPanel);
    rebuildTree();
}

private void rebuildTree() {
    nodeMap.clear();
    walk(root, null);                              // reindex id → Node
    mapPanel.render(treeToJson(root));               // push to client → D3 redraws
}

private void handleNodeAction(String id, String action) {
    Node n = nodeMap.get(id);
    switch (action) {
        case "click"     -> openEditorDialog(n);
        case "addRule"   -> { ((Composite) n).children.add(new Rule(…)); rebuildTree(); }
        case "addGroup"  -> { ((Composite) n).children.add(new Composite(…)); rebuildTree(); }
    }
}

The pattern works because the model is the single source of truth. The JS never mutates anything — it just renders what Java tells it and reports clicks back. Any edit (operator change, new rule, delete) updates the Java model and triggers rebuildTree(), which serialises a fresh JSON and ships it across. The SVG is torn down and rebuilt; D3's join machinery makes that cheap. No diff plumbing on either side. tree.add(root); // .pe-tree styles draw the connector lines

The recursion is straightforward: every composite's children get their own UnorderedList appended to the composite's ListItem, and the CSS takes care of the visual hierarchy. The "+ Rule" / "+ Group" affordances sit inline next to each composite as dashed-outline buttons (see buildComposite()).

§ 15Things to remember

  • The kit's job is to render kit-styled DOM and expose a fluent setter API. Anything beyond that (data binding, presenters, services) belongs to views.
  • Real Vaadin field when users type or you need keyboard accessibility (auth, reply bar). Kit field when the screen is a placeholder or visual prop.
  • NativeButton for icon buttons, tiles, menu rows. Browsers give us native focus, keyboard activation, ARIA role for free.
  • Lazy header / lazy slot — never attach an empty wrapper. Only when content is set does the wrapper appear in the DOM. LkPage, LkCard, LkField, LkBanner all use this.
  • Overridden add() is how surfaces (LkCard, LkAuthCard, LkPage, PolicyGroup) route children to a body slot without you having to type .body.add(…) at the call site.
  • Static factories (LkSearchPanel.defaults, LkNotifPanel.buyer, LkAccountMenu.admin) are great for reference content. When the real data feed arrives, replace the call — the constructor still takes generic data.
  • Toasts is the only allowed notification surface. Don't call Notification.show. The wrapper guarantees consistent styling and the always-present dismiss button.
  • Icon vocabulary is closed. Adding a new glyph means adding a path-string pair to the LkIcon.PATHS map. Unknown names render empty — if a button looks bare, check the icon name.
  • Component over Div in cells. LkGrid accepts any Component as a cell value, so don't flatten badges to strings. Drop the LkBadge directly.
