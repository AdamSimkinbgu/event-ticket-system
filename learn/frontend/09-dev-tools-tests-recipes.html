Dev tools, tests & end-to-end recipes

The dev panel for flipping personas without re-authenticating, the smoke tests that catch kit-API misuse, and five recipes for shipping new things — from a new view to a fresh Java↔JS bridge.

§ 1The DevPanel — persona flipping for free

src/main/java/com/ticketing/system/Presentation/dev/

A floating debug widget attached to every UI by a Spring bean. Sits in the bottom-right corner as a "BUG · DEV" pill. Clicking it opens a Dialog that lets you mutate MockAuth, MockCompanies, MockSession, and MockPermissions directly — bypassing the sign-in form.

Two files

FileRole
DevPanel.javaThe dialog itself + the trigger button. ~370 LOC of pure UI.
DevPanelInitializer.javaSpring bean implementing VaadinServiceInitListener — attaches the trigger pill to every UI on init.

The five sections inside the dialog

SectionWhat it lets you do
Persona shortcutsSix toggle pills: Guest, Member, Manager, Co-owner, Founder, Admin. Each pill reflects live session state — selected = filled black, deselected = white outline. Click to toggle. Stack any combination (e.g. Founder + Manager + Admin all on at once).
IdentityTextField for display name + two checkboxes (Signed in, Admin pool). Direct knobs on the underlying MockAuth attributes.
CompaniesPer-company rows with "Make current" + "Remove" buttons + a row of "+ Founder / + Co-owner / + Manager" add buttons. Maps directly to MockCompanies.add(…).
Manager permissionsOnly renders when the current company role is "Manager". Six checkboxes (one per MockPermissions.GRANTABLE capability). Toggle to grant/revoke.
Capability inspectorRead-only. Renders Capabilities.forCurrentUser() as a cloud of LkBadges. Recomputed every time the dialog opens.

How persona stacking works

Each persona pill encodes a question about session state, not a fixed assignment:

PillSelected whenClick toggles
Guest!isSignedIn() && !isAdmin()Sign out + clear cart + clear companies
MemberisSignedIn()Sign in (default "adam") OR sign out if pure member with no other roles
Manager / Co-owner / FounderHas a company with that roleAdd a seeded company with that role OR remove all companies of that role
AdminisAdmin()MockAuth.setAdmin(!isAdmin())

Because the flags are independent (see 05 §2), Founder + Manager + Admin can all be active simultaneously. The capability inspector at the bottom of the dialog confirms the union live.

The attach pattern

@org.springframework.stereotype.Component
public class DevPanelInitializer implements VaadinServiceInitListener {
    @Override
    public void serviceInit(ServiceInitEvent event) {
        event.getSource().addUIInitListener(uiInit ->
            uiInit.getUI().add(DevPanel.trigger())
        );
    }
}

The trigger pill is added to every UI as a floating element. It outlives view navigation because it's added to the UI, not to a layout. Same Spring-bean pattern as AuthBootstrap — remove @Component and the dev panel silently disappears.

In production, hide this. Today the dev panel ships on every UI in every environment. When we approach release we'll guard the initializer with a Spring profile check (@Profile("!prod")) so it only attaches in development. The capability mutations it exposes are not auth bypass-safe.

§ 2VaadinSmokeTest — 8 cheap checks

src/test/java/com/ticketing/system/unit/presentation/VaadinSmokeTest.java

A Spring Boot test that boots the application context but doesn't render anything in a browser. Eight tests, each catching a different class of bug at near-zero cost.

What it checks

TestCatches
contextLoadsCleanlySpring + Vaadin can boot at all. The auth bootstrap doesn't throw during init.
vaadinClassesAreOnClasspathVaadin Flow types are reachable — UI, AppLayout, VaadinService, the Spring servlet.
presentationPackageIsConfiguredRoutes live under com.ticketing.system.Presentation so Vaadin's scanner finds them.
layoutsAreLoadableMainLayout and WorkspaceLayout compile and resolve without missing imports.
coreViewsInstantiateSpot-checks BrowseEventsView + GlobalHistoryView. If the kit breaks, this catches the failure for two representative views.
platformAdminViewsInstantiateEvery PlatformAdminLayout view (admin dashboard, announcements, complaint queue, global history, org tree) constructs without exceptions.
kitComponentsInstantiateLkIcon, LkBtn, LkCard, LkBadge, VkSeat, VkSeatLegend — the kit primitives every view depends on.
toastsUtilityIsLoadableToasts.class resolves. The "is this static utility on the classpath?" check.

What it deliberately does NOT check

  • Rendering. Vaadin's UI tests require a UI context (a real UI instance bound to a Vaadin session), which SpringBootTest doesn't spin up. We'd need vaadin-testbench or a headless browser.
  • Navigation. The smoke test never fires a BeforeEnterEvent, so it never exercises AuthBootstrap.
  • Business logic. That's the job of application-layer acceptance tests — not the presentation tier.
Why is "spot check two views" enough? Because every view extends LkPage and uses the same kit. If the kit compiles and two representative views instantiate, the rest are almost certainly fine. The actual bugs that slip past the smoke test are bad enum references and wrong API calls in views the smoke test doesn't spot-check — which is why we added platformAdminViewsInstantiate after the LkSelect API trap shipped in AdminAnnouncementsView.

§ 3CapabilitiesTest — 3 resolver assertions

src/test/java/com/ticketing/system/unit/presentation/CapabilitiesTest.java

A plain JUnit 5 test (no Spring) that exercises the capability resolver without a Vaadin session. Capabilities.forCurrentUser() short-circuits to GUEST_BASE when there's no session — the test asserts that baseline plus the helper trio (has / hasAll / hasAny) is consistent. 

@Test
void guestBaseIncludesCatalogButNotCheckout() {
    Set<Capability> caps = Capabilities.forCurrentUser();
    assertTrue (caps.contains(Capability.BROWSE_CATALOG));
    assertFalse(caps.contains(Capability.CHECKOUT));
    assertFalse(caps.contains(Capability.OWNER_WORKSPACE));
    assertFalse(caps.contains(Capability.ADMIN_WORKSPACE));
}

These three tests aren't exhaustive — they assert the guest baseline only. The signed-in / owner / admin bundles are exercised through the smoke test's view-instantiation checks, which transitively touch every cap a view declares.

§ 4The test pyramid for this codebase

The V2 spec contains the line "בדיקות אינן נדרשות עבור ממשק המשתמש" — UI tests are not required. That's a deliberate scope decision. The tests we do have, and the tests we deliberately don't:

LayerWhat runsWhy this much
Application services (Core/Application)Full acceptance tests via BaseDomainTest. Each UC gets a test class.Business correctness lives here. Every UC has invariants that must hold.
Domain (Core/Domain)Invariant tests integrated into the application-layer tests via track().Per feedback memory: invariants attach to existing tests rather than living in isolated XInvariantTest files.
Presentation kit + viewsVaadinSmokeTest + CapabilitiesTest. 11 assertions total. Boot the context, instantiate a sample, no rendering.Cheap canary — if the kit breaks compile, every view fails to instantiate. UI rendering tests are explicitly out of scope.
JS bridge (D3 policy editor)None.The Java side is exercised by smoke; the JS side renders only in a browser. If we ever ship for real, that's when we add vaadin-testbench or Playwright.

Result: the test suite is small, fast (3 seconds), and catches the bugs you would actually ship. The cost of not having UI tests is that you discover layout bugs by looking at the running app — which is the right discovery channel for visual problems anyway.

Five recipes for things you'll actually need to do. Each is the full transcript — every file you touch, every line you add, in the order it has to happen.

RECIPE 1Add a new view from scratch

We'll add a "Saved events" view for members — a list of favourited events.

Step 1 — Create the view class

src/main/java/com/ticketing/system/Presentation/views/account/SavedEventsView.java
@Route(value = "saved-events", layout = MainLayout.class)
@PageTitle("Saved events · TicketHub")
@PermitAll
public class SavedEventsView extends LkPage {

    public SavedEventsView() {
        title("Saved events");
        subtitle("Events you've bookmarked for later.");

        // Empty state — first thing to ship.
        LkCard empty = new LkCard().pad(40);
        empty.add(Lk.h3("No saved events yet"));
        empty.add(Lk.muted("Tap the ☆ on any event to add it here."));
        empty.add(new LkBtn("Browse events").variant(LkBtn.Variant.primary)
            .onClick(e -> UI.getCurrent().navigate(BrowseEventsView.class)));
        add(empty);
    }
}

Step 2 — Wire the avatar menu

In MainLayout.buildMemberMenu, add a menu item: 

new LkMenu.Item("star", "Saved events").onClick(() -> UI.getCurrent().navigate(SavedEventsView.class)),

Step 3 — Add a smoke check

In VaadinSmokeTest: 

import com.ticketing.system.Presentation.views.account.SavedEventsView;

@Test
void savedEventsInstantiates() {
    assertDoesNotThrow(SavedEventsView::new,
        "SavedEventsView failed to construct");
}

Step 4 — Run the smoke test

./mvnw test -Dtest=VaadinSmokeTest -o

Done. The view is reachable at /saved-events, discoverable from the avatar menu, and the smoke test catches future kit-API breakage.

Suppose we want managers to be able to export company sales CSV.

Step 1 — Add the enum value

Step 2 — Add it to the right bundle in Capabilities.java

Manager-grantable means: in the manager grant set, also unconditionally in the owner bundle. 

private static final Set<Capability> OWNER_BUNDLE = unmodifiable(EnumSet.of(

public static final Set<Capability> GRANTABLE = EnumSet.of(

Founders/co-owners always see the button. Managers see it only if granted.

RECIPE 3Add a Java ↔ JS bridge

The template lifted straight from PolicyTreeMap. Let's sketch a "venue heatmap" component that renders ticket-sales density as an SVG heatmap.

Step 1 — Write the JS module

src/main/frontend/venue-heatmap.js
const D3_URL = 'https://cdn.jsdelivr.net/npm/d3@7/+esm';
let d3Promise = null;
function getD3() {
    if (!d3Promise) d3Promise = import(/* @vite-ignore */ D3_URL);
    return d3Promise;
}

window.renderVenueHeatmap = async function(container, dataJson) {
    const d3 = await getD3();
    const data = JSON.parse(dataJson);
    container.innerHTML = '';
    // … d3 rendering …
    nodes.on('click', (ev, d) => {
        if (container.$server) container.$server.onSeatClicked(d.seatId);
    });

Step 2 — Write the Java wrapper

Presentation/components/venue/VenueHeatmap.java
@JsModule("./venue-heatmap.js")
public class VenueHeatmap extends Div {

    private Consumer<String> onSeatClick;
    private String pendingJson;

    public VenueHeatmap() {
        addClassName("venue-heatmap");
        getStyle().set("height", "520px");
        addAttachListener(e -> { if (pendingJson != null) callRender(pendingJson); });
    }

    public void render(String dataJson) {
        if (isAttached()) callRender(dataJson); else pendingJson = dataJson;
    }

    private void callRender(String dataJson) {
        getElement().executeJs("window.renderVenueHeatmap(this, $0);", dataJson);
    }

    public void setOnSeatClicked(Consumer<String> handler) { onSeatClick = handler; }

    @ClientCallable
    private void onSeatClicked(String seatId) {
        if (onSeatClick != null) onSeatClick.accept(seatId);
    }
}

That's the whole pattern. Same five pieces as PolicyTreeMap: dynamic D3 import, window.renderXxx, @JsModule, executeJs, @ClientCallable. Reuse for org tree drag-drop, seat-map zoom, real-time notification feeds.

RECIPE 4Swap a mock for a real service

When V2-RES-04 lands, MockCart becomes ReservationService. Here's the shape of that swap.

Before

// view code
int count = MockCart.size();
List<MockCart.Item> items = MockCart.getItems();
MockCart.add(new MockCart.Item(…));

After

// view code — UNCHANGED
int count = MockCart.size();
List<MockCart.Item> items = MockCart.getItems();
MockCart.add(new MockCart.Item(…));

// MockCart.java body changes — but the surface stays
public final class MockCart {

    @Autowired
    private static ReservationService reservationService;

    public static int size() {
        return reservationService.viewMyCart(currentUserId()).size();
    }

    public static List<Item> getItems() {
        return reservationService.viewMyCart(currentUserId())
            .stream().map(MockCart::toViewItem).toList();
    }

    // …
}

The trick is that MockCart stays as the API surface even after the swap. View code never changes. Only the body of the static methods does. Rename the file later if "Mock" becomes inaccurate — that's find/replace, not a refactor.

The same pattern works for MockAuth, MockCompanies, MockPermissions

Each becomes a thin facade over a Spring-injected service. The surface stays; the body delegates to the real implementation. The capability resolver, the dev panel, every view — all keep working.

RECIPE 5Add a brand-new role bundle

Suppose we want a "platform auditor" role — can view global sales but nothing else.

Step 1 — Add the capabilities

Capability.java
// Platform auditor
AUDITOR_WORKSPACE,
VIEW_AUDIT_TRAIL,
EXPORT_AUDIT_CSV

Step 2 — Add the bundle in Capabilities.java

private static final Set<Capability> AUDITOR_BUNDLE = unmodifiable(EnumSet.of(
    AUDITOR_WORKSPACE, VIEW_AUDIT_TRAIL, EXPORT_AUDIT_CSV
));

Step 3 — Add the flag to MockAuth.java

private static final String AUDITOR_KEY = "mockAuth.auditor";

public static boolean isAuditor() {
    VaadinSession s = VaadinSession.getCurrent();
    return s != null && Boolean.TRUE.equals(s.getAttribute(AUDITOR_KEY));
}
public static void setAuditor(boolean on) {
    VaadinSession s = VaadinSession.getCurrent();
    if (s != null) s.setAttribute(AUDITOR_KEY, on);
}

Step 4 — Plug into the resolver

public static Set<Capability> forCurrentUser() {
    Set<Capability> caps = EnumSet.copyOf(GUEST_BASE);

    if (MockAuth.isAdmin())    caps.addAll(ADMIN_BUNDLE);
    if (MockAuth.isAuditor())  caps.addAll(AUDITOR_BUNDLE);   // new

    // … rest unchanged
}

Step 5 — Update AuthBootstrap's fallback table

private static final Set<Capability> AUDITOR_CAPS = EnumSet.of(
    AUDITOR_WORKSPACE, VIEW_AUDIT_TRAIL, EXPORT_AUDIT_CSV
);

private static Class<? extends Component> fallbackFor(Capability c) {
    if (ADMIN_CAPS.contains(c))    return LoginView.class;
    if (AUDITOR_CAPS.contains(c))  return LoginView.class;   // or a dedicated landing
    if (WORKSPACE_CAPS.contains(c)) …
    return MyCompaniesView.class;
}

Step 6 — Add an Auditor toggle to the dev panel

Same pattern as the existing Admin toggle — one more pill in the persona row, reads MockAuth.isAuditor(), writes MockAuth.setAuditor(bool). Then write a view annotated @RequireCapability(AUDITOR_WORKSPACE) — AuthBootstrap will gate it automatically.

Steps 1–6 are the entire workflow. Notice what didn't change: no view modifications, no layout edits, no RequireCapability annotation update. The capability layer absorbs the new role and every existing consumer (views, drawer filters, conditional render branches) keeps working unchanged.
