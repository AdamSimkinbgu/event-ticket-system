Layouts, routing & the capability layer

How the three shells fit together — and how the {@code Capability} enum + a one-line annotation determines what each persona is allowed to see and do.

§ 1What a layout actually is

A Vaadin layout is just a regular component that has been told "every view that asks for me as its parent gets rendered inside me". Concretely, a layout is a Java class that extends AppLayout and implements RouterLayout. The Vaadin router uses that pair as a hint: when it builds the DOM for a route, it walks up from the leaf view, finds the first RouterLayout ancestor, and inserts the view into that ancestor's content slot.

The mental model

A page on screen is a chain of nested components: 

+ <body>
  + MainLayout              // extends AppLayout
    + LkTopBar              // shell — the blue bar across the top
    + <router-outlet>       // where the view lands
      + BrowseEventsView    // extends LkPage (a Div)
        + filter chips
        + grid of cards

Every route's rendered output is "the layout plus the view stuffed inside". That's why we have layouts in the first place — so the topbar and side-nav don't have to be rewritten on every single screen.

Vaadin's AppLayout

com.vaadin.flow.component.applayout.AppLayout is a built-in component with three slots:

  • Navbar — the strip across the top. We fill this with LkTopBar.
  • Drawer — the optional left panel. We fill this with LkSideNav for owner / admin.
  • Content — the area where the routed view renders. We never touch it directly.

The Java API for the slots is addToNavbar(component), addToDrawer(component). That's what every layout in this codebase calls.

§ 2The @Route annotation

A view becomes reachable in the browser by declaring annotations at the class level: 

@Route(value = "owner/managers", layout = WorkspaceLayout.class)
@PageTitle("Managers · TicketHub")
@PermitAll
@RequireCapability(Capability.APPOINT_MANAGER)
public class ManagerListView extends LkPage {
    // ...
}
AnnotationWhat it doesRequired?
@Route(value, layout)Maps a URL pattern to this view, and says which layout wraps it.Yes
@PageTitle(text)Sets the browser tab title.No (defaults to class name)
@AnonymousAllowedAnyone, signed-in or not, may enter.One of the two auth annotations
@PermitAllAny signed-in user may enter (subject to capability gates).One of the two
@RequireCapability(Capability.X)Fine-grained RBAC — the user must hold capability X or they're forwarded to a fallback view.Optional — only when you want a per-view gate beyond just "signed in"

URL templates

Route values can include parameter placeholders prefixed with :: 

@Route(value = "event/:eventId", layout = MainLayout.class)
@Route(value = "owner/events/:eventId", layout = WorkspaceLayout.class)
@Route(value = "receipt/:orderId", layout = MainLayout.class)

The view receives those params via BeforeEnterEvent.getRouteParameters().get("eventId") inside beforeEnter(). This is how event-detail and receipt URLs stay deep-linkable.

How layout = stitches in

When Vaadin matches the URL, it walks: what layout did this view's @Route ask for? Render that layout. Insert the view into its content slot. Everything in the layout's constructor (topbar, drawer, side-nav) is already in place by the time the view mounts.

§ 3MainLayout — the buyer / guest shell

src/main/java/com/ticketing/system/Presentation/layouts/MainLayout.java

The simplest of the three. No drawer — just a topbar. Used by every @Route that names MainLayout.class as its parent.

The topbar rebuild

private void rebuildTopBar(String activeLabel) {
    if (topBar != null) topBar.getElement().removeFromParent();

    boolean signedIn = MockAuth.isSignedIn();
    String name = signedIn ? MockAuth.displayName() : "Guest";

    List<LkTopBar.NavItem> nav = new ArrayList<>();
    nav.add(new LkTopBar.NavItem("Browse",     BrowseEventsView.class));
    nav.add(new LkTopBar.NavItem("My Tickets", MyAccountView.class));
    // Admin tab appears for anyone who holds ADMIN_WORKSPACE (admin alone OR admin + member).
    if (Capabilities.has(Capability.ADMIN_WORKSPACE)) {
        nav.add(new LkTopBar.NavItem("Admin", AdminDashboardView.class));
    }

    LkTopBar bar = new LkTopBar(LkTopBar.Variant.MAIN)
        .brand("TicketHub", LandingView.class)
        .nav(nav, activeLabel)
        .searchDefault()
        .cart(CartView.class, MockCart.size(), shortestCartDeadlineMs())
        .bellDefault(false);

    if (signedIn) {
        bar.account(initials(name), name, buildMemberMenu(name));
    } else {
        bar.guestActions(LoginView.class, RegisterView.class);
    }

    topBar = bar;
    addToNavbar(topBar);
}

Three places this layout consults capabilities (all live on Capabilities.has(…)):

  • Nav: Admin tab. Shown only when ADMIN_WORKSPACE is held. Lets an admin who is browsing as a member jump back into the platform shell with one click.
  • Avatar menu: Workspace vs. Become organizer. The menu item swaps based on OWNER_WORKSPACE (held) vs. REGISTER_COMPANY (held but no company yet).
  • Right slot: account vs. guest actions. Driven by signed-in state (which itself implies MEMBER_* capabilities).

The avatar menu's conditional

if (Capabilities.has(Capability.OWNER_WORKSPACE)) {
    menu.add(new LkMenu.Item("building", "Workspace")
        .onClick(() -> UI.getCurrent().navigate(OwnerDashboardView.class)));
} else if (Capabilities.has(Capability.REGISTER_COMPANY)) {
    menu.add(new LkMenu.Item("plus", "Become an organizer")
        .hint("free")
        .onClick(() -> UI.getCurrent().navigate(CompanyRegistrationView.class)));
}

Notice that both branches still exist in the codebase — we don't delete "Become an organizer" if you're an owner. We just don't add it to the menu in that branch. The menu rebuilds on every navigation, so flipping the state (registering a company) shows the correct item the very next time you click around.

Why rebuild on every navigation?

  1. The topbar reflects state that can change between page loads (cart size, signed-in name, current capability set). Building it once would miss any subsequent flip.
  2. The active-nav highlight is per-route. findActiveLabel(event) walks the active component chain to find a class in the NAV_LABELS map — that's how CartView and CheckoutView both highlight "My Tickets" (they're grouped under one tab).

private void rebuildDrawer(String activeLabel) {

    ownerNav = new LkSideNav("Owner").items(visible, activeLabel);
    drawerWrap.add(ownerNav);

§ 6The navigation lifecycle

Vaadin gives you two observer interfaces to hook into:

InterfaceFiresUsed by
BeforeEnterObserverBefore the view's constructor runs — you can event.forwardTo(…) to redirect.AuthBootstrap — the global guard.
AfterNavigationObserverAfter the view is in the DOM — you can read the active route.All three layouts — for topbar / drawer state.

Why our layouts use afterNavigation

At afterNavigation time, the active component chain is already known. We use event.getActiveChain() to find which view is showing and update the active highlight: 

private static String findActiveLabel(AfterNavigationEvent event) {
    for (HasElement el : event.getActiveChain()) {
        String label = NAV_LABELS.get(el.getClass());
        if (label != null) return label;
    }
    return null;
}

The "chain" starts at the leaf view (e.g. BrowseEventsView) and walks up to the layout. The first class in the chain that exists in NAV_LABELS wins. That's how CartView, CheckoutView, OrderConfirmationView all highlight the same "My Tickets" tab — they're all explicitly mapped to that label.

Why beforeEnter is where RBAC lives

beforeEnter fires before any view code runs. That's the only safe place to redirect away from a forbidden route — if you waited until afterNavigation, the view's constructor would have already done its work (queried services, drawn UI). The gate has to slam shut before that happens.

§ 7Why RBAC needs to be a kit concern

A common mistake is to put role checks inside each view's constructor: 

// ANTI-PATTERN — don't do this
public CompanyEventListView() {
    if (!Capabilities.has(Capability.VIEW_COMPANY_EVENTS)) {
        Toasts.failure("You don't have permission.");
        UI.getCurrent().navigate(OwnerDashboardView.class);
        return;
    }
    // … build the UI
}

Three problems:

  1. Spreads the rule across 33 views. If you ever change the policy ("managers can also cancel events"), you have to find every site.
  2. The view runs anyway. The constructor body executes, the UI is partly built, you then navigate away. Side effects (service calls in the constructor) still happen.
  3. No safety net. A new view written by a teammate who forgets the check is silently public.

The fix is to push RBAC to two places, neither of them the view body:

  1. A declarative annotation on the view class. Easy to grep, hard to forget.
  2. A central guard (a BeforeEnterListener) that fires before any view code runs and inspects the annotation.

§ 8The four personas — and how they stack

guest

Anonymous visitor

No session. Can browse the landing page, the event catalog, event details, and the seat preview. Can't reach checkout, can't see "My …" anywhere.
member

Signed-in user

Has MockAuth.isSignedIn() == true. Adds: cart, checkout, my account, profile, invitations, support inbox, submit complaint. Sees "Become an organizer" CTA unless they already have a company.
owner

Member with a company role

A flavor of member — OWNER_WORKSPACE is held because MockSession.currentCompany() != null. The actual workspace caps depend on the role (Founder / Co-owner / Manager).
admin

Platform administrator

MockAuth.isAdmin() is an independent flag. An admin can also be signed in as a member — the two persona families stack.

Stacking

The crucial change vs. the old marker system: admin and member personas stack. A single session can be all of:

  • Signed in (member baseline)
  • Founder of company A
  • Co-owner of company B
  • Manager of company C with a specific grant set
  • Plus admin

The capability set is just the union of all the bundles each persona brings. The DevPanel debug widget exposes this directly — you can toggle Manager, Co-owner, Founder, and Admin independently and see the capability cloud rebuild.

§ 9The capability layer

RBAC in this codebase is built on three Java files plus two session stores:

FileRole
security/Capability.javaThe closed enum of every gate-able action. ~28 values across guest / member / workspace / founder / admin bundles.
security/Capabilities.javaThe resolver. Reads {@code MockAuth + MockSession + MockCompanies + MockPermissions} and returns the {@code Set<Capability>} for the current session. {@code has()} / {@code hasAll()} / {@code hasAny()} helpers.
security/RequireCapability.javaRuntime annotation gating a whole view by one capability. Read by {@code AuthBootstrap}.

The Capability enum, organised by bundle

public enum Capability {
    // Universal (guest can have these)
    BROWSE_CATALOG, VIEW_EVENT_DETAILS, VIEW_SEAT_MAP, USE_CART,

    // Member (signed in, any persona)
    CHECKOUT, MEMBER_ACCOUNT, SUBMIT_COMPLAINT,
    REGISTER_COMPANY, MANAGE_INVITATIONS,

    // Workspace access (any company role)
    OWNER_WORKSPACE,

    // Manager-grantable (Founders + Co-owners hold these unconditionally)
    VIEW_COMPANY_EVENTS, EDIT_COMPANY_EVENTS,
    VIEW_COMPANY_SALES, EDIT_PURCHASE_POLICIES,
    RESPOND_INQUIRIES, MANAGE_VENUE_MAPS,

    // Owner-only (Founder + Co-owner; not grantable to managers)
    APPOINT_MANAGER, APPOINT_CO_OWNER,
    EDIT_MANAGER_PERMISSIONS, REVOKE_MANAGER, CANCEL_EVENT,

    // Founder-only
    DISSOLVE_COMPANY, TRANSFER_FOUNDERSHIP,

    // Platform admin
    ADMIN_WORKSPACE, VIEW_GLOBAL_HISTORY, BROADCAST_ANNOUNCEMENT,
    MANAGE_COMPLAINTS, VIEW_ORG_TREES, ADMIN_SETTINGS
}

The resolver

Capabilities.forCurrentUser() is the single point that maps session state to a Set<Capability>. The structure is straightforward additive composition: 

public static Set<Capability> forCurrentUser() {
    Set<Capability> caps = EnumSet.copyOf(GUEST_BASE);

    // Admin caps stack independently — an admin can also be a member.
    if (MockAuth.isAdmin()) {
        caps.addAll(ADMIN_BUNDLE);
    }

    if (!MockAuth.isSignedIn()) return caps;

    caps.addAll(MEMBER_BASE);

    MockCompanies.Company current = MockSession.currentCompany();
    if (current == null) return caps;

    caps.add(Capability.OWNER_WORKSPACE);
    switch (current.role()) {
        case "Founder" -> {
            caps.addAll(OWNER_BUNDLE);
            caps.addAll(FOUNDER_EXTRA);
        }
        case "Co-owner" -> caps.addAll(OWNER_BUNDLE);
        case "Manager" -> caps.addAll(MockPermissions.forCompany(current.id()));
    }
    return caps;
}

Read top-to-bottom: guest baseline, optional admin stack, optional member stack, optional workspace stack. No conditionals about which persona — just additive bundles.

The @RequireCapability annotation

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface RequireCapability {
    Capability value();
}

A view opts in by annotating its class: 

@Route(value = "owner/policies", layout = WorkspaceLayout.class)
@PermitAll
@RequireCapability(Capability.EDIT_PURCHASE_POLICIES)
public class PurchasePolicyEditorView extends LkPage {
    // …
}

That's it. AuthBootstrap reads the annotation and forwards lacking users to a sensible fallback (see § 10).

Why annotation, not interface

Earlier versions used marker interfaces (RequiresOwnerCompany, RequiresAdminRole). They were collapsed into the capability annotation because:

  • One mechanism. Views no longer need both a marker and (eventually) an annotation for the same protection.
  • Annotation carries data. The Capability value is the rule. A marker would just say "this view is protected" — the actual rule had to live elsewhere.
  • One fallback table. AuthBootstrap.fallbackFor(Capability) picks the right redirect target for the missing cap. With markers, every gate had its own hardcoded redirect.

For per-section checks, just call the resolver

@RequireCapability gates entire views. For finer-grained UI — hide a section, disable a button — use Capabilities.has(…) directly inside the view body. The EventManagementView danger zone is a textbook example: 

add(buildDetailsForm());
add(buildLinkedEditorsCard());
add(buildStatusCard());
if (Capabilities.has(Capability.CANCEL_EVENT)) {
    add(buildDangerCard());     // founder / co-owner only
}

Same for drawer items, dashboard tiles, row actions, …

The pattern repeats: list the candidates, filter against Capabilities.has(…), render what survives. WorkspaceLayout filters its drawer this way (§ 4). OwnerDashboardView filters its tile grid the same way — a manager with only inquiries permission sees a two-tile dashboard, not seven tiles four of which would bounce off the gate when clicked.

§ 10AuthBootstrap — the global guard

src/main/java/com/ticketing/system/Presentation/security/AuthBootstrap.java

Spring bean (@Component) implementing VaadinServiceInitListener. Vaadin auto-discovers any such bean and registers it during service initialization. We use that hook to attach a BeforeEnterListener to every UI instance. 

@org.springframework.stereotype.Component
public class AuthBootstrap implements VaadinServiceInitListener {

    @Override
    public void serviceInit(ServiceInitEvent event) {
        event.getSource().addUIInitListener(uiInit ->
            uiInit.getUI().addBeforeEnterListener(this::guard));
    }
}

The two-step gate

private void guard(BeforeEnterEvent event) {
    Class<?> target = event.getNavigationTarget();
    RequireCapability cap = target.getAnnotation(RequireCapability.class);

    // 1. Auth gate.
    if (!target.isAnnotationPresent(AnonymousAllowed.class)) {
        if (!MockAuth.isSignedIn()) {
            event.forwardTo(LoginView.class);
            return;
        }
    }

    // 2. Capability gate.
    if (cap != null && !Capabilities.has(cap.value())) {
        event.forwardTo(fallbackFor(cap.value()));
    }
}

The fallback table

When the user lacks a capability, fallbackFor(Capability) picks the right place to send them — not just one hardcoded destination. The logic:

Missing capabilityForward toRationale
Admin cap (ADMIN_WORKSPACE, VIEW_GLOBAL_HISTORY, …)LoginViewMember is hitting the admin shell — bounce to sign-in. Re-auth as admin lands on the platform shell.
Workspace cap when user has no companyCompanyRegistrationViewThey need to become an owner first.
Workspace cap when user already has a companyOwnerDashboardViewThey're inside the workspace but lack this specific permission — park them on the home where they can see what they CAN do.
Anything elseMyCompaniesViewSafe default.

The @Component annotation is what makes this work. Putting @Component on the class is what makes Vaadin call it during init. If you remove the annotation, the gate silently stops working. The smoke test wouldn't catch that — it never tries to navigate. So if you ever see views that should be gated rendering unauthenticated, the first thing to check is the @Component annotation.

§ 11The unified sign-in surface

The codebase has two "you've hit a wall, please authenticate" surfaces. Both reached via routes in views/auth/:

EndpointClassReached when…
/loginLoginViewAuth gate fires for a non-anonymous view. Single sign-in surface for both admins and members.
/registerRegisterViewUser clicks "Register" link. Refuses admin usernames so the pools stay disjoint at registration.
/register-companyCompanyRegistrationViewWorkspace cap gate fires for a signed-in member who has no company yet.

The username decides the pool

The unified LoginView picks which session API to call based on whether the entered username is in the admin pool: 

private void attemptSignIn(TextField username, PasswordField password) {
    if (MockAuth.isAdminUsername(username.getValue())) {
        MockAuth.signInAsAdmin(username.getValue());
        Toasts.success("Signed in as admin · " + username.getValue());
        UI.getCurrent().navigate(AdminDashboardView.class);
    } else {
        MockAuth.signIn(username.getValue());
        Toasts.success("Signed in as " + username.getValue());
        UI.getCurrent().navigate(BrowseEventsView.class);
    }
}

The pool of admin usernames lives in MockAuth.ADMIN_USERNAMES — a closed Set<String>. RegisterView consults the same set to refuse admin names at sign-up time.

Why one form is enough

The earlier design had two distinct sign-in endpoints (/login and /admin/sign-in) to make the admin surface "hidden". The AdminLoginView file is gone now — the pools are still disjoint at registration, but the sign-in form itself is shared. This simplifies routing (AuthBootstrap only has one "please sign in" target) and means the smart fallback in § 10 always knows where to send people.

Admin + member is now a real session shape. An admin who is also a member sees the buyer shell plus an "Admin" tab in the topbar (see § 3). They don't need to sign out and back in to switch contexts — one click jumps from the member shell into the platform shell.

§ 12Tailoring the UI by capability

The gate stops a user from reaching a view. But you also want to not show buttons they couldn't use. Four patterns:

1. Conditional avatar menu items

The avatar menu in MainLayout reads capabilities directly: 

if (Capabilities.has(Capability.OWNER_WORKSPACE)) {
    menu.add(new LkMenu.Item("building", "Workspace").onClick(…));
} else if (Capabilities.has(Capability.REGISTER_COMPANY)) {
    menu.add(new LkMenu.Item("plus", "Become an organizer").hint("free").onClick(…));
}

2. Conditional topbar nav tabs

Same file, just above — the Admin tab appears only when the corresponding capability is held: 

nav.add(new LkTopBar.NavItem("Browse",     BrowseEventsView.class));
nav.add(new LkTopBar.NavItem("My Tickets", MyAccountView.class));
if (Capabilities.has(Capability.ADMIN_WORKSPACE)) {
    nav.add(new LkTopBar.NavItem("Admin", AdminDashboardView.class));
}

3. Conditional drawer items

WorkspaceLayout pairs each drawer item with the capability that gates it (§ 4) and filters per navigation. A manager with limited grants sees a shorter drawer; a founder sees the whole list.

4. Conditional content inside views

The dashboard tile grid filters by capability so clicks always go somewhere useful: 

if (Capabilities.has(Capability.VIEW_COMPANY_EVENTS))
    tiles.add(tile("calendar", "My Events",    …, CompanyEventListView.class));

if (Capabilities.has(Capability.RESPOND_INQUIRIES))
    tiles.add(tile("comment",  "Member Inquiries", …, CompanyInquiryInboxView.class));

if (Capabilities.has(Capability.APPOINT_MANAGER))
    tiles.add(tile("users",    "Managers",     …, ManagerListView.class));

Same pattern works for buttons inside views, row-action icons, etc. The EventManagementView danger zone is owner-only: 

col.add(buildLinkedEditorsCard(), buildStatusCard());
if (Capabilities.has(Capability.CANCEL_EVENT)) {
    col.add(buildDangerCard());
}

What about hidden vs. omitted?

Don't use setVisible(false) for RBAC. Two reasons:

  1. The DOM is still there. A determined user can flip the display:none in dev tools and reveal the button. The click handler is still active.
  2. It runs the constructor. The button is still allocated, its click handler is still registered. Any service look-ups inside the button's configuration still execute.

Skipping the add() entirely is cheaper, safer, and doesn't require the client to be trusted. If you ever need setVisible, it's because you want to toggle at runtime (e.g. as the user types in a form), not because of a role.

§ 13Capability cookbook

EX 1Add a new workspace view

Say you want a "Compliance dashboard" available only to founders. Pick a capability and gate it:

@Route(value = "owner/compliance", layout = WorkspaceLayout.class)   // 1. Routes under workspace shell
@PageTitle("Compliance · TicketHub")
@PermitAll                                                            // 2. Any signed-in user can try

EX 4Add a new manager-grantable permission

Suppose you want a new manager permission for "Export sales CSV". Three steps: 

// 1. Add the enum value

    …
    EXPORT_COMPANY_SALES_CSV,    // new
    …

// 2. Add it to MockPermissions.GRANTABLE
public static final Set<Capability> GRANTABLE = EnumSet.of(

Founders and co-owners get it automatically (it's in OWNER_BUNDLE). Managers get it only if their company's grant set includes it. The dev panel will show the new checkbox in the Manager permissions section without any other change.

EX 5Multi-company / multi-role debugging

The DevPanel floating widget (bottom-right BUG pill, attached to every UI by DevPanelInitializer) lets you stack personas freely. EX 6Guest-only CTA

The landing page wants a "Sign up free" hero button only when there's no signed-in user. A member visiting the landing page should see "Browse events" instead: 

if (MockAuth.isSignedIn()) {
    heroCta.add(new LkBtn("Browse events").variant(LkBtn.Variant.primary)
        .onClick(e -> UI.getCurrent().navigate(BrowseEventsView.class)));
} else {
    heroCta.add(
        new LkBtn("Sign up free").variant(LkBtn.Variant.primary)
            .onClick(e -> UI.getCurrent().navigate(RegisterView.class)),
        new LkBtn("Browse without signing up").variant(LkBtn.Variant.tertiary)
            .onClick(e -> UI.getCurrent().navigate(BrowseEventsView.class))
    );
}

For state checks that don't fit neatly into a capability (like "are you a guest?"), it's fine to read MockAuth directly. Capabilities cover actions; identity distinctions like signed-in / not are cleaner with the flag.

§ 14Pitfalls + things to remember

Pitfall 1 — Forgetting @AnonymousAllowed on a public view

LoginView, RegisterView, LandingView, and BrowseEventsView are all annotated @AnonymousAllowed. Without that, the auth gate would forward a guest to LoginView… which itself requires auth, infinite redirect, browser shows an error.

Pitfall 2 — @RequireCapability without @PermitAll

The capability gate runs after the auth gate. If the auth gate rejects the user first (because the view doesn't have @PermitAll or @AnonymousAllowed), the capability check never runs. In practice, gated views should always carry @PermitAll — the capability annotation is the actual gate, while @PermitAll just satisfies Vaadin's built-in security.

Pitfall 3 — Don't put RBAC in the constructor body

Once AuthBootstrap exists, putting a role check inside a view's constructor is both redundant and wrong — the constructor body runs before UI.getCurrent().navigate takes effect, so any side effect already happened. If you must check inside a view, do it from BeforeEnterObserver on the view itself.

Pitfall 4 — Drawer items aren't a security boundary

The drawer filter hides items the user can't use. But that's a UX choice, not a security guarantee. If a user somehow learns the URL of a gated view and types it directly, they hit the @RequireCapability gate and forward to the fallback. The gate is the real boundary; the drawer is a discoverability convenience.

Pitfall 5 — Adding a capability without choosing a bundle

When you add an enum value to Capability, decide where it lives in Capabilities.forCurrentUser() — is it part of OWNER_BUNDLE? FOUNDER_EXTRA? MockPermissions.GRANTABLE (manager-grantable)? An ADMIN_BUNDLE cap? Without picking a home, the new value never gets added to any session's capability set — Capabilities.has(NEW) will always return false and the corresponding UI just… never appears. Easy to miss in code review.

Things to remember

  • Layout = shell + slot. Pick the right layout in @Route(layout=…) and the shell renders automatically.
  • Three layouts: MainLayout for buyer/guest, WorkspaceLayout for the workspace (any company role), PlatformAdminLayout for system admin.
  • RBAC is a kit concern — @RequireCapability on the class + Capabilities.has(…) inside the body. Never view constructors.
  • One mechanism: the capability annotation replaced the earlier marker interfaces. RequiresOwnerCompany and RequiresAdminRole no longer exist.
  • Two-step gate in AuthBootstrap: auth → capability. Fallback table picks the redirect.
  • Personas stack. Admin and member are independent flags. A single session can be a Founder of A + Manager of B + Admin all at once.
  • One sign-in form. LoginView decides admin vs member by username. RegisterView refuses admin names. The earlier AdminLoginView is gone.
  • Skip add() for capability-driven visibility, never setVisible(false).
  • The DevPanel is your friend — toggle personas, see capabilities live, reload to see the rest of the app re-render under the new shape.
