The security/ folder, file by file

Five files, ~250 LOC of Java, the entire RBAC machinery. We cover what each does, why it's shaped that way, and what each becomes when V2-F-03 swaps the mocks for Spring Security.

§ 1The folder at a glance

FileLOCRole
MockAuth.java~95Session-scoped mock authentication state. Three flags on VaadinSession: signed-in, name, admin. Plus the closed admin-username set.
Capability.java~50The enum — 28 values across 5 bundles.
Capabilities.java~120The resolver. Maps the combined Mock state to a Set<Capability>.
RequireCapability.java~35Runtime annotation. A single Capability value().
AuthBootstrap.java~115Spring bean. Attaches a global BeforeEnterListener to every UI. Runs the auth + capability gate.

03-layouts-and-routing covered how these fit together at the conceptual level. This page covers implementation — the session keys, the bundle composition, the fallback table, the disjoint-pool enforcement, and the swap plan.

§ 2securityMockAuth

A namespace of static methods backed by three VaadinSession attributes. No instances, no state outside the session.

The three session keys

private static final String SIGNED_IN_KEY = "mockAuth.signedIn";
private static final String NAME_KEY      = "mockAuth.name";
private static final String ADMIN_KEY     = "mockAuth.admin";

Three independent flags — this is important. An earlier version conflated them into a single role string ("member" or "admin"). That collapsed when the dev panel needed to model an admin who is also a member — you can't have one role be both. Splitting signedIn and admin into independent flags solved it.

The public API, end to end

MethodMutatesNotes
signIn(name)SIGNED_IN_KEY = true, NAME_KEY = nameLeaves ADMIN_KEY untouched. Member-style sign-in.
signInAsAdmin(name)All three: signed-in + admin both trueAdmin does imply signed-in (it's convenient to read isSignedIn() as "session active at all").
setAdmin(boolean)ADMIN_KEY; if true also auto-signs in with default name "admin"The dev-panel knob. Sets admin independently.
signOut()All three to null/falseReturns to guest.
isSignedIn() / isAdmin() / displayName()ReadsReturn false/null when no VaadinSession (unit-test friendly).

The closed admin-username pool

public static final Set<String> ADMIN_USERNAMES = Set.of(
    "admin", "platform.admin", "bar.miyara"
);

public static boolean isAdminUsername(String name) {
    return name != null && ADMIN_USERNAMES.contains(name.trim().toLowerCase());
}

The disjoint-pool rule is enforced at three call sites:

  • LoginView reads the username and decides which API to call (member signIn vs signInAsAdmin). Same form, two outcomes.
  • RegisterView refuses admin names at sign-up. No member account can ever shadow an admin name.
  • The dev panel ignores the pool entirely — it's a dev tool, you can be admin + member + founder + manager all at once. The pool guard lives only on the production sign-in surfaces.
Why a closed set instead of a database table? This is mock code. Real admin provisioning is the I.1.4 / V2-F-03 territory — a system_admin table seeded by the DefaultSystemAdmin initializer. The mock skips all that and hardcodes the names. When the real auth lands, ADMIN_USERNAMES is the only thing that needs deleting.

§ 3securityCapability

A plain Java enum. The whole interface: 

public enum Capability {

    // Universal (guest can have these)
    BROWSE_CATALOG, VIEW_EVENT_DETAILS, VIEW_SEAT_MAP, USE_CART,

    // Member (signed in)
    CHECKOUT, MEMBER_ACCOUNT, SUBMIT_COMPLAINT,
    REGISTER_COMPANY, MANAGE_INVITATIONS,

    // Workspace access
    OWNER_WORKSPACE,

    // Manager-grantable (also held by Founder + Co-owner)
    VIEW_COMPANY_EVENTS, EDIT_COMPANY_EVENTS,
    VIEW_COMPANY_SALES, EDIT_PURCHASE_POLICIES,
    RESPOND_INQUIRIES, MANAGE_VENUE_MAPS,

    // Owner-only (not grantable to managers)
    APPOINT_MANAGER, APPOINT_CO_OWNER,
    EDIT_MANAGER_PERMISSIONS, REVOKE_MANAGER, CANCEL_EVENT,

    // Founder-only
    DISSOLVE_COMPANY, TRANSFER_FOUNDERSHIP,

    // Platform admin
    ADMIN_WORKSPACE, VIEW_GLOBAL_HISTORY, BROADCAST_ANNOUNCEMENT,
    MANAGE_COMPLAINTS, VIEW_ORG_TREES, ADMIN_SETTINGS
}

Why the bundles are commented in the enum itself

The comments aren't decoration — they're telling future you which bundle in Capabilities.java needs updating when you add a value. Add a new universal cap? Update GUEST_BASE. Owner-only? Update OWNER_BUNDLE. Forget which bundle? The comment above the value tells you.

The "silent failure" trap. Adding a value to the enum and forgetting to put it in a bundle means Capabilities.has(NEW_VALUE) always returns false. The UI just… never shows the corresponding button or view. The smoke test won't catch it (no view references the new value yet). Easy to ship by mistake. The grep test: after adding a value to the enum, grep Capabilities.java for it — if it doesn't appear in a bundle, your wiring is broken.

§ 4securityCapabilities

The resolver. The single place that turns "what does the session look like right now?" into "what is the user allowed to do?".

Bundle definitions

// Closed sets, unmodifiable, computed at class load.
private static final Set<Capability> GUEST_BASE  = EnumSet.of(BROWSE_CATALOG, …);
private static final Set<Capability> MEMBER_BASE = EnumSet.of(CHECKOUT, …);
private static final Set<Capability> OWNER_BUNDLE = EnumSet.of(VIEW_COMPANY_EVENTS, …);
private static final Set<Capability> FOUNDER_EXTRA = EnumSet.of(DISSOLVE_COMPANY, …);
private static final Set<Capability> ADMIN_BUNDLE = EnumSet.of(ADMIN_WORKSPACE, …);

EnumSet is the right choice here — backed by a bitvector, so union/intersect are O(1) and the whole set fits in a long.

The resolver itself

public static Set<Capability> forCurrentUser() {
    Set<Capability> caps = EnumSet.copyOf(GUEST_BASE);

    // Admin caps stack independently — an admin can also be a member.
    if (MockAuth.isAdmin()) {
        caps.addAll(ADMIN_BUNDLE);
    }

    if (!MockAuth.isSignedIn()) return caps;

    caps.addAll(MEMBER_BASE);

    MockCompanies.Company current = MockSession.currentCompany();
    if (current == null) return caps;

    caps.add(Capability.OWNER_WORKSPACE);
    switch (current.role()) {
        case "Founder" -> {
            caps.addAll(OWNER_BUNDLE);
            caps.addAll(FOUNDER_EXTRA);
        }
        case "Co-owner" -> caps.addAll(OWNER_BUNDLE);
        case "Manager" -> caps.addAll(MockPermissions.forCompany(current.id()));
    }
    return caps;
}

Read top to bottom — the logic is purely additive

  1. Start with the guest baseline (everyone gets the catalog).
  2. Admin caps stack on top (independent of member state — this is the change that enabled persona stacking).
  3. Not signed in? Return now. Admins who are not signed in get just ADMIN_BUNDLE + GUEST_BASE, which is the right answer for that edge case.
  4. Signed in: add member baseline.
  5. Has a current company? Add OWNER_WORKSPACE and the role-specific bundle. Managers get whatever MockPermissions says.

The helper trio

public static boolean has(Capability c)            { return forCurrentUser().contains(c); }
public static boolean hasAll(Capability... cs)     { // every cap in cs must be present }
public static boolean hasAny(Capability... cs)     { // at least one }

99% of view code calls has(…). hasAll / hasAny are there for the occasional gate that needs multiple caps. None of these cache — the resolver runs end-to-end on each call. It's cheap (a couple of EnumSet unions) so caching would be premature.

Why no cache? Two reasons. (1) The resolver reads VaadinSession attributes — if we cached and the dev panel mutated the session, the cache would lie. (2) The cost is genuinely tiny. Premature caching is the source of "why is this stale?" bugs. Re-compute every time.

§ 5securityRequireCapability

A type-level annotation. The whole interface: 

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface RequireCapability {
    Capability value();
}

Three details worth noticing

  • @Retention(RUNTIME) — otherwise AuthBootstrap can't read it via reflection at BeforeEnterEvent time. Java's default retention is CLASS, which is invisible to reflection.
  • @Target(TYPE) — only valid on classes. Forces the annotation onto views, never on methods.
  • Capability value() — the implicit value() attribute lets callers write @RequireCapability(Capability.X) instead of @RequireCapability(value = Capability.X). Annotation usability.

Usage at the call site

@Route(value = "owner/managers", layout = WorkspaceLayout.class)
@PermitAll
@RequireCapability(Capability.APPOINT_MANAGER)
public class ManagerListView extends LkPage {
    // …
}

That's it. AuthBootstrap reads the annotation, calls Capabilities.has(…), forwards on miss.

§ 6securityAuthBootstrap

The hot loop — runs on every navigation event in every UI in every session. Despite that, the file is ~115 lines because all the heavy thinking moved into the resolver.

The bean discovery dance

@org.springframework.stereotype.Component
public class AuthBootstrap implements VaadinServiceInitListener {

    @Override
    public void serviceInit(ServiceInitEvent event) {
        event.getSource().addUIInitListener(uiInit ->
            uiInit.getUI().addBeforeEnterListener(this::guard));
    }
}

Three nested registration steps:

  1. @Component — Spring discovers the bean during context init.
  2. VaadinServiceInitListener — Vaadin finds the Spring bean and calls serviceInit once per Vaadin service (per servlet, basically per app).
  3. addUIInitListener — the inner lambda runs every time a new UI is created. Inside that, we attach guard as a BeforeEnterListener on that specific UI.
If you remove @Component, the gate silently disappears. Spring won't register the bean, Vaadin won't find it, no BeforeEnterListener will be attached, every view becomes public. The smoke test won't catch this — it never navigates. This is the single most fragile annotation in the codebase.

The two-step gate

private void guard(BeforeEnterEvent event) {
    Class<?> target = event.getNavigationTarget();
    RequireCapability cap = target.getAnnotation(RequireCapability.class);

    // 1. Auth gate — every non-anonymous view needs a session.
    if (!target.isAnnotationPresent(AnonymousAllowed.class)) {
        if (!MockAuth.isSignedIn()) {
            event.forwardTo(LoginView.class);
            return;
        }
    }

    // 2. Capability gate.
    if (cap != null && !Capabilities.has(cap.value())) {
        event.forwardTo(fallbackFor(cap.value()));
    }
}

The fallback table — where to send people on a miss

This is the part 03 only sketched. The full logic: 

private static Class<? extends Component> fallbackFor(Capability c) {
    if (ADMIN_CAPS.contains(c)) {
        // Member tried to reach the admin shell — bounce to login,
        // re-auth as admin lands on the platform shell.
        return LoginView.class;
    }
    if (WORKSPACE_CAPS.contains(c)) {
        // No company yet → become one. Has a company but lacks this
        // specific permission → park on the workspace home.
        return Capabilities.has(Capability.OWNER_WORKSPACE)
            ? OwnerDashboardView.class
            : CompanyRegistrationView.class;
    }
    return MyCompaniesView.class;
}

The two static lookup sets — why they live here, not in Capability

ADMIN_CAPS and WORKSPACE_CAPS are EnumSets declared at the top of AuthBootstrap. They list which capabilities belong to which fallback category. Why here and not on the enum?

  • The enum stays pure data. No knowledge of fallback routing.
  • If we ever add a second consumer of these groupings (say, a deny-list audit log), it gets its own copy without polluting Capability.
  • The fallback policy is a presentation concern — lives next to the code that uses it.

EX 1Add a new fallback category

Suppose you add an "auditor" persona that can view sales globally but nothing else. Auditor views fall back to a dedicated AuditorHomeView: 

private static final Set<Capability> AUDITOR_CAPS = EnumSet.of(
    Capability.VIEW_GLOBAL_HISTORY    // + future auditor caps
);

private static Class<? extends Component> fallbackFor(Capability c) {
    if (ADMIN_CAPS.contains(c))    return LoginView.class;
    if (AUDITOR_CAPS.contains(c))  return AuditorHomeView.class;
    if (WORKSPACE_CAPS.contains(c)) return Capabilities.has(OWNER_WORKSPACE) ? … : …;
    return MyCompaniesView.class;
}

Order matters: check the most specific category first, fall through to the default last.

§ 7The V2-F-03 swap plan

The whole point of building this layer in front of mocks is that it survives the swap to real auth. Here's what each file becomes when V2-F-03 (#209) lands Spring Security.

FileWhat staysWhat changes
Capability.javaEverything. The enum is the contract.Nothing. New values may be added; existing ones don't move.
RequireCapability.javaEverything. Views keep declaring it.Nothing.
Capabilities.javaBundle composition, helper signatures.Reads SecurityContextHolder.getContext().getAuthentication() + a real CompanyMembershipService instead of MockAuth/MockCompanies. Bundles map to GrantedAuthority sets.
AuthBootstrap.javaThe before-enter gate + fallback table.The auth check becomes "is there a non-anonymous Authentication in the security context?" instead of MockAuth.isSignedIn(). The rest stays.
MockAuth.javaNothing.Deleted. Replaced by Spring Security primitives + AuthenticationService.

The view-side promise

After the swap, not a single view changes. Capabilities.has(X) still works. @RequireCapability(X) still works. The drawer filter in WorkspaceLayout, the dashboard tile filter in OwnerDashboardView, the conditional avatar items in MainLayout — all keep working. The only diffs are in Capabilities.java's implementation body.

That's the value of this layer. Mocks today, real auth tomorrow, view code unchanged either way.

The DevPanel survives too. It mutates MockAuth directly today. After V2-F-03, it'll mutate the Spring Security context via a privileged setAuthentication helper that's only exposed in non-production profiles. Same dev experience, real auth model underneath.

§ 8Things to remember

  • Three independent flags in MockAuth: signed-in, name, admin. Admin is independent of signed-in by deliberate choice — persona stacking depends on it.
  • The closed admin-username pool (ADMIN_USERNAMES) is enforced at LoginView + RegisterView, ignored by the dev panel.
  • The enum is the contract. Adding a value but forgetting to put it in a bundle is the most common silent failure — the cap never shows up. Always grep the bundles after adding an enum value.
  • The resolver runs end-to-end on every call. Don't cache. The cost is tiny and stale caches are the source of "why won't this update?" bugs.
  • @Component on AuthBootstrap is load-bearing. Remove it and every gated view becomes public, silently.
  • Fallback table lives in AuthBootstrap, not the enum. The enum is data; routing is presentation.
  • After V2-F-03, only Capabilities.java + AuthBootstrap.java change body; everything else (the enum, the annotation, every view) stays untouched. That's the value of the abstraction.
