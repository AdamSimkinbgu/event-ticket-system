The session/ folder, file by file

Four files, ~230 LOC, all share the same backing pattern: state in VaadinSession attributes. Together they model the cart, company memberships, the currently-focused company, and per-manager permission grants.

§ 1Orientation — the VaadinSession-as-cache pattern

All four mocks share the same shape: static utility class, private constructor, public methods that read/write VaadinSession.getCurrent() attributes by string key. Once you grok the pattern, all four read the same way.

The pattern

public final class MockXxx {

    private static final String KEY = "mockXxx.thing";

    private MockXxx() { }

    @SuppressWarnings("unchecked")
    public static SomeType get() {
        VaadinSession s = VaadinSession.getCurrent();
        if (s == null) return defaultValue();   // unit-test friendly
        SomeType v = (SomeType) s.getAttribute(KEY);
        if (v == null) {
            v = newDefault();
            s.setAttribute(KEY, v);
        }
        return v;
    }

    public static void set(SomeType v) {
        VaadinSession s = VaadinSession.getCurrent();
        if (s != null) s.setAttribute(KEY, v);
    }
}

Five things this pattern guarantees:

  1. Per-user storage. VaadinSession.getCurrent() returns the session for the current user — different browser tabs in the same browser share it; different users get different sessions.
  2. Survives page reloads. Session attributes live on the server, indexed by the session cookie. F5 doesn't reset state.
  3. Unit-test friendly. No VaadinSession.getCurrent()? Returns a sensible default. CapabilitiesTest exploits this to assert the guest baseline without booting a UI.
  4. No instances ever. Private constructor + final class. Java's idiom for "this is a namespace, not a type".
  5. Lazy initialization. The default value is computed on first access — for collections, this means "first reader gets an empty mutable list backed by the session".
Why VaadinSession, not HttpSession? VaadinSession wraps HttpSession and adds Vaadin-aware locking (important when push or async UI updates are involved). It also gives you access from any thread that holds the Vaadin lock. Reading HttpSession directly would skip the lock and risk races. Always use VaadinSession from Vaadin code.

§ 2sessionMockCart

The buyer's reservation cart. Holds line items (each with a per-line expiry heldUntil) and a separate global "checkout deadline" used by CheckoutView.

The Item record

public record Item(
    String  title,
    String  seat,
    int     priceCents,
    Instant heldUntil,    // per-line TTL, set when added
    String  gradient     // CSS gradient string keyed to category
) { }

priceCents is an int on purpose — integer arithmetic in cents avoids the floating-point trap of "$1.10 + $1.10 = $2.1999999…". Display layer formats as a decimal.

Two session attributes

private static final String KEY               = "mockCart.items";
private static final String CHECKOUT_DEADLINE = "mockCart.checkoutDeadline";

The items list and the checkout deadline are independent attributes. The deadline is set on entry to CheckoutView (10 minutes from now) and cleared when the cart empties.

API

MethodNotes
getItems()Returns the live list. Mutations on the returned List are persistent — that's the point.
add(Item) / remove(Item)Convenience. remove also clears the global deadline when the cart goes empty.
clear()Empties the list + clears the deadline.
size()Used by the navbar cart badge.
subtotalCents()Sum of priceCents over all items.
getCheckoutDeadline() / setCheckoutDeadline(Instant) / clearCheckoutDeadline()The global timer.

The interplay with the navbar timer chip

MainLayout.shortestCartDeadlineMs() takes the union of every line's heldUntil AND the global checkoutDeadline, picks the minimum still in the future, and feeds it to LkTopBar.cart(…) as the timer chip's endpoint. So once CheckoutView sets a 10-minute global deadline, the chip in the navbar follows whichever expires first (usually the line items, since they've been ticking longer).

Why a global deadline AND per-line ones? Because the spec (V2-CHECK-01) wants entry to checkout to refresh all line timers into a single unified countdown. Today the mock keeps both — the global one is set on checkout entry, the line ones decay independently. When V2-RES-04 wires real reservations, the model will collapse to one deadline per cart line + one global checkout window.

§ 3sessionMockCompanies

The list of companies the current user has a role in. The role field on each Company is what drives the capability resolver.

The Company record

public record Company(
    String id,
    String name,
    String description,
    String contactEmail,
    String role,          // "Founder" / "Co-owner" / "Manager"
    String status,        // "Active"
    int    members,
    int    activeEvents
) { }

The same user can have entries with different roles — the dev panel uses this to model a Founder of one company who is also a Manager somewhere else. The capability resolver picks the role of whichever company is currently selected (see MockSession).

API

MethodNotes
forCurrentUser()The live mutable list. Returns an empty (but persistent) list on first read.
isOwner()Misleading name. Returns true if the list is non-empty — a manager passes this check too. Kept for historical reasons; capability code uses Capabilities.has(OWNER_WORKSPACE) instead.
size()Number of companies the user has any role in.
add(Company) / clear()Mutations. Used by CompanyRegistrationView and the dev panel.
The isOwner() name is a historical leak. Originally the only role modelled was Founder, so "has companies" meant "is owner". When managers were added, the name became inaccurate — a manager also passes isOwner(). Don't use it for new code. Capabilities.has(Capability.OWNER_WORKSPACE) is the correct check.

§ 4sessionMockSession

A single piece of state — the id of the currently-selected company — that drives which role's capability bundle gets applied. Without it, a user who founds two companies wouldn't have a way to switch between them.

API

MethodNotes
currentCompanyId()The stored id, or null if none set.
setCurrentCompany(String id)Persist the user's choice.
clearCurrentCompany()On sign-out, persona switch, etc.
currentCompany()The Company record. Falls back to the first company in MockCompanies.forCurrentUser() when no explicit selection is stored — "first one wins" is the simplest sensible default.

Why a separate class

We could've stored "current company" as an attribute inside MockCompanies. We didn't because:

  • Different lifetime. The selection survives even when the list mutates around it (as long as the selected id still exists).
  • Different consumer. The dashboard subtitle, the venue editor's scope, the policy editor's "applies to" pill — they all want the selection, not the list. Letting them depend on MockSession directly is cleaner than having them all reach through MockCompanies.
  • Different swap target. When V2 hits, the list moves to CompanyMembershipService (one query) while the selection might stay on the UI side as a session preference. Splitting them makes the swap easier.

§ 5sessionMockPermissions

Per-company manager grants. Founders and co-owners hold the whole OWNER_BUNDLE unconditionally — only managers need explicit grants.

The closed grantable set

public static final Set<Capability> GRANTABLE = EnumSet.of(
    Capability.VIEW_COMPANY_EVENTS,
    Capability.EDIT_COMPANY_EVENTS,
    Capability.VIEW_COMPANY_SALES,
    Capability.EDIT_PURCHASE_POLICIES,
    Capability.RESPOND_INQUIRIES,
    Capability.MANAGE_VENUE_MAPS
);

This is the explicit allow-list of what an owner can grant a manager. Caps outside the set (APPOINT_CO_OWNER, CANCEL_EVENT, DISSOLVE_COMPANY) are owner-only by design — the grant(String, Capability) method enforces this: 

public static void grant(String companyId, Capability c) {
    if (!GRANTABLE.contains(c))
        throw new IllegalArgumentException("Not grantable to managers: " + c);
    forCompany(companyId).add(c);
}

A bug in calling code — trying to grant a manager CANCEL_EVENT — fails fast with a useful exception message, not silently.

The default grant for a fresh manager

private static final Set<Capability> DEFAULT_GRANTS = EnumSet.of(
    Capability.VIEW_COMPANY_EVENTS,
    Capability.RESPOND_INQUIRIES
);

First time anyone queries forCompany(id) for a company with no stored grant set, the manager gets this baseline: can see events and respond to inquiries, can't edit anything. Owners can grant(…) more later.

API

MethodNotes
forCompany(id) / forCurrentCompany()Returns the live grant set for the company. Lazy-initialises to DEFAULT_GRANTS.
grant(id, cap) / revoke(id, cap)Single-cap mutations. grant enforces GRANTABLE.
setAll(id, set)Replace the whole grant set. Filters out non-GRANTABLE values silently.

One key per company

VaadinSession attributes are keyed by string. MockPermissions uses "mockPermissions." + companyId as the key, so a user who is a Manager of three companies has three independent grant sets stored under three keys.

Why no global "all grants" view? Because the resolver doesn't need it. Capabilities only ever asks about the current company. The other grant sets exist but are invisible until you switch companies. That's deliberate — the mental model "what can I do in this company" is simpler than "what can I do anywhere".

§ 6How the four fit together

The dependency graph between the four mocks plus MockAuth (covered in 05): 

  MockAuth     ──┐
                 │
  MockCompanies ─┼──► Capabilities.forCurrentUser()
                 │         │
  MockSession ───┤         ├──► Capabilities.has(X)  ──► every view check
                 │         │
  MockPermissions┘         └──► AuthBootstrap.guard() ──► every navigation

  MockCart ───────────────────► MainLayout shortest-cart-deadline ──► navbar timer chip
                                CheckoutView line/global deadlines
                                CartView per-line countdowns

Three takeaways:

  1. MockCart is independent. It doesn't affect capabilities — cart state is "what tickets are reserved" not "what can the user do."
  2. The other four feed Capabilities. Auth + which companies + which is current + per-manager grants combine into one capability set.
  3. Every consumer reads through Capabilities. No view bypasses the resolver to read mocks directly. That keeps the swap path clean.

What each mock becomes when the corresponding V2 wiring lands.

The view-side promise (again)

Same as security: no view code changes. Every read goes through Capabilities.has(…) + MockCart.size() etc.; the resolver already abstracts where the data comes from. The swap is changing the body of Capabilities.forCurrentUser() to call services instead of mocks.

EX 1Watch the cart timer update across reloads

Open /event/coldplay/seats, select two seats, click "Hold for 10 min". Hit F5. The cart icon in the navbar still shows the badge and the timer keeps counting down — because the items are in VaadinSession, not in a JavaScript variable. Open a new tab to /cart — same state, same countdown. Now open a new browser (or incognito), navigate to the same URL: empty cart, no timer. That's the per-user scoping at work.

EX 2Use the dev panel to flip Manager grants live

Open the dev panel. Set persona to Manager. The Manager-permissions section shows the six GRANTABLE capabilities as checkboxes — two pre-checked. Toggle EDIT_PURCHASE_POLICIES on, click Apply & reload. Now navigate to /owner/policies — you can reach it. Toggle it off, reload, navigate again — the capability gate forwards you to the dashboard. The drawer entry for "Purchase Policies" appears and disappears with the toggle.

§ 8Things to remember

  • One pattern across all four. Static class, private constructor, VaadinSession-attribute storage by string key. Once you read one, you can read all four.
  • Use VaadinSession, never HttpSession directly. The Vaadin wrapper handles locking; the raw API doesn't.
  • Items + global deadline are independent in MockCart. The navbar timer chip picks the minimum of both.
  • The same user can have different roles in different companies. MockSession picks which company is "current" and drives which role's capability bundle applies.
  • Managers get explicit grants. Founders + co-owners hold the whole bundle unconditionally. GRANTABLE is the closed allow-list of what an owner can hand to a manager.
  • MockCompanies.isOwner() is misleading. Use Capabilities.has(OWNER_WORKSPACE) instead.
  • Three of four swap out, one stays. Cart, companies, permissions all become real services. MockSession (current-company selection) stays as a UI preference.
  • Resolver doesn't cache. Reads everything fresh on every has(…). Premature caching of session data was a self-inflicted-wound source.
