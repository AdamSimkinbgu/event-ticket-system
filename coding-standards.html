1. About this manual

This document is the source of truth for "how we write code in this repo". Every convention here came from one of three sources: the BGU SE Workshop lectures (V0 and V1 prep), the V2 web-application-framework lecture, or real PRs we shipped (notably #201 — the AOP / IRepository infrastructure rework — and #202 / #205 — the polymorphic zone + reservation refactors).

When you write Java in this repo, you will: define classes in DDD aggregates, expose them through application services, persist them through a Memory*Repository, and (in V2) wire a Vaadin view + presenter on top. Every layer has rules. This manual lists them with examples and explains why.

How to use this doc

  • Read top-to-bottom on your first day, even if you only write code in one layer.
  • Reference specific sections during PR review (link to the anchor).
  • If you find a convention here that's wrong or missing, update the doc in the same PR that motivated the change.
  • This is a living document. Conventions change as the project matures; check the git log for recent updates.
WHYWhy have a coding standards doc at all? Because the cost of reviewing a PR depends on how much of it surprises you. If every contributor follows the same conventions, reviews become cheap ("does it compile? are tests green?") instead of expensive ("is this the right pattern? would Adam disagree?"). At seven contributors with rotating PR reviewers, the same disagreement gets re-litigated weekly without a shared document.

2. Domain-Driven Design — the philosophy

DDD is the central technical philosophy of this project (lecture 1's framing). The domain is the heart of the system — and it must be isolated from database, UI, and communication concerns. This is the hexagonal architecture: domain in the center, everything else pointing inward.

LECTURELecture 1, Hexagonal Architecture: "Your domain must be isolated — agnostic to DB, UI, and communication layers." The domain holds the rules. Everything else is plumbing.

2.1 Aggregates — the unit of consistency

An aggregate is a cluster of objects that share a consistency boundary. All operations that need to keep multiple objects in sync go through one entry point — the aggregate root. Anything outside the aggregate can only touch the root, never the internal entities.

This codebase has 12 aggregates:

  • User · Session · Event · Ticket · ActiveOrder · OrderReceipt
  • ProductionCompany · Admin · Notification · Conversation
  • Lottery (model-only) · EventQueue (model-only)

2.2 The «Aggregate root» IXxxRepository naming convention

Course staff at BGU mandated "one entry point per aggregate". To satisfy that, the team uses a fused naming: the interface IXxxRepository serves two roles in the white class diagram — (a) it is the aggregate root marker, and (b) it is the repository contract. The «Aggregate root» stereotype is drawn on the same box.

Domain — IUserRepository.java
package Core.Domain.users;

import Core.Domain.shared.IRepository;
import java.util.Optional;

/**
 * Aggregate root for the User aggregate.
 *
 * Repository contract + aggregate-entry marker fused into one interface,
 * per the course-mandated "one entry point per aggregate" rule.
 */
public interface IUserRepository extends IRepository<User, Integer> {

    Optional<User> findByUsername(String username);

    Optional<User> findByEmail(String email);

    boolean existsByUsername(String username);
}
WHYTwo roles fused: the diagram only has room for one box per aggregate. The course requires the «Aggregate root» stereotype and forbids redundant repository entries. Combining them into IXxxRepository with the «Aggregate root» stereotype satisfies both rules. Don't propose splitting them in future reviews.

2.3 No cross-aggregate relations on the diagram

This is the golden rule from lecture 1. Two aggregates never share a line on the white class diagram. Cross-aggregate references in code are by ID only:

Cross-aggregate refs — by ID, never by object
// GOOD - Ticket holds the user's ID, not a User pointer.
public class Ticket {
    private final int holderUserId;     // cross-aggregate ref
    private final int eventId;          // cross-aggregate ref
    private final String zoneId;
    private final String seatLabel;     // nullable for standing zones
    // ...
}

// BAD - never do this.
public class Ticket {
    private final User holder;          // <-- crosses aggregate boundary
    private final Event event;          // <-- ditto
    // ...
}
WHYThree reasons (from lecture): (1) separation of concerns — each aggregate stays a self-contained unit you can reason about; (2) DB locking efficiency — loading a ticket doesn't drag a user and event into the transaction; (3) scalability — aggregates can live on different servers / DBs.

When a use case crosses aggregates, lecture 1 taught two patterns. Only these two are taught — anything else is suspect.

  • Coordinating Service (with optional Domain Service) — strong consistency, tight coupling. The application service retrieves both aggregates and orchestrates. If the logic is heavy, extract a «Domain Service» (different from an Application Service — domain services hold business logic that spans entities but doesn't fit any one of them).
  • Domain Events (publisher / subscriber) — eventual consistency, weak coupling, scales well. One service publishes; another listens.

2.4 The white class diagram — what it shows

The "white" diagram is a skeleton. It shows classes, their stereotypes, and the most valuable relations (multiplicity + role). It does not show attributes or methods — those live in code.

  • Submit only Domain + Application layers (plus repository interfaces). Infrastructure is excluded.
  • Value Objects are usually omitted unless they carry a structural concept the team needs to anchor on.
  • State machines appear as «Enum» boxes connected to the entity, not as attributes.
  • No sequence, state, or activity diagrams as deliverables. Behavior decisions belong in prose.

2.5 Anti-pattern — "dumb puppet show run by God Services"

Application services are orchestrators, not the place for business logic. Pulling all logic into services and leaving entities anemic is the explicit anti-pattern lecture 1 warned against. Logic lives in the domain.

BAD Service does everything; entity is a bag of getters: 
// BAD - business rule sitting in the service
public void confirmReservation(int eventId, int userId, List<String> seats) {
    Event event = eventRepo.findById(eventId).orElseThrow();
    if (event.getStatus() != EventStatus.OPEN) throw new IllegalStateException();
    for (String label : seats) {
        Seat seat = event.getVenue().getZone("A").getSeats().get(label);
        if (seat.getStatus() != SeatStatus.AVAILABLE) throw new IllegalStateException();
        seat.setStatus(SeatStatus.RESERVED);
        seat.setReservedBy(userId);
        seat.setReservedAt(Instant.now());
    }
    eventRepo.save(event);
}
GOOD Service orchestrates; entity enforces the rule: 
// GOOD - the zone knows how to reserve itself
public void reserve(int eventId, int userId, InventorySelection selection) {
    eventRepo.lockForUpdate(eventId);
    try {
        Event event = eventRepo.findById(eventId).orElseThrow();
        event.reserveFor(userId, selection);    // domain method, throws if invalid
        eventRepo.save(event);
    } finally {
        eventRepo.unlock(eventId);
    }
}

3. Layered architecture

Four layers. Dependency points inward — outer layers depend on inner, never the reverse.

LayerWhat goes hereKnows about
Presentation V2 NEW Vaadin views + presenters, REST controllers, security config, MainLayout, custom components Application (calls services), DTOs only
Application Services (one method per UC), external ports (interfaces) Domain (repositories + aggregates), DTOs
Domain Aggregate roots, entities, value objects, enums, IXxxRepository contracts Nothing outside itself. No Spring annotations, no Vaadin, no JPA.
Infrastructure Memory*Repository impls, port adapters (Jwt, Bcrypt, Stub*), RepositoryLocks, LoggingAspect, scheduling Domain (implements its interfaces), Application (implements its ports)
WHY The dependency direction matters. If Domain depended on Infrastructure, you couldn't unit-test the domain without Spring + a database. By inverting the dependency (Domain defines IUserRepository; Infrastructure implements it as MemoryUserRepository), the domain is testable in pure Java. The same trick lets V3 swap Memory for JPA without changing Domain or Application.

4.1 Package rules

  • One package per aggregate in Domain. Don't mix two aggregates in one package.
  • No upward package dependencies. Domain never imports from Application, Infrastructure, or Presentation.
  • Tests mirror the package structure under src/test/java.
  • Shared cross-cutting code in Core/Domain/shared/ only for things every aggregate uses (IRepository, InvariantChecked, InventorySelection).

5.2 The InvariantChecked interface

Every aggregate root and every Value Object that carries structural invariants implements InvariantChecked. The method checkInvariants() throws if the object is in a broken state. It's called automatically after every test that tracks the aggregate (see section 11).

Domain — InvariantChecked.java
package Core.Domain.shared;

/**
 * Implemented by every aggregate root and every VO that carries structural
 * invariants. The method must throw a runtime exception (typically
 * IllegalStateException) when the object's state violates an invariant.
 *
 * BaseDomainTest calls this automatically after every test that wraps the
 * object with track(...) — see V2-LOG-01 and feedback_invariant_test_integration.
 */
public interface InvariantChecked {
    void checkInvariants();
}
Example — User.checkInvariants()
@Override
public void checkInvariants() {
    if (id <= 0)
        throw new IllegalStateException("User.id must be positive, was " + id);
    if (username == null || username.isBlank())
        throw new IllegalStateException("User.username must not be blank");
    if (passwordHash == null || passwordHash.isBlank())
        throw new IllegalStateException("User.passwordHash must not be blank");
    if (email == null || !email.contains("@"))
        throw new IllegalStateException("User.email is malformed: " + email);
    for (CompanyAppointment appt : appointments) {
        appt.checkInvariants();          // recurse into owned sub-entities
    }
}
WHY Invariants enforced through structure, not tests. Lecture 1: "validity constraints should be enforced via system structure and operations, not via tests." The constructor + every mutator inside the entity should leave it in a valid state. checkInvariants() is the safety net that catches any code path that violates the rules under concurrent operations.

5.3 Composite pattern — purchase policies

Lecturer explicitly recommended Composite for purchase / discount policies. For Tier C we build only the purchase half. The pattern lets us evaluate arbitrarily nested AND/OR expressions through a single interface.

Domain — PurchasePolicy (Composite)
public interface PurchasePolicy {
    PolicyResult evaluate(BuyerContext buyer, CartContext cart);
}

// Leaves -----------------------------------------------------------

public final class AgeAtLeast implements PurchasePolicy {
    private final int minAge;
    public AgeAtLeast(int minAge) { this.minAge = minAge; }
    @Override public PolicyResult evaluate(BuyerContext b, CartContext c) {
        return b.age() >= minAge
            ? PolicyResult.ok()
            : PolicyResult.reject("Tickets are restricted to age " + minAge + " and over");
    }
}

public final class QuantityAtMost implements PurchasePolicy { /* ... */ }
public final class QuantityAtLeast implements PurchasePolicy { /* ... */ }

// Composites -------------------------------------------------------

public final class AndPolicy implements PurchasePolicy {
    private final List<PurchasePolicy> children;
    public AndPolicy(List<PurchasePolicy> children) {
        this.children = List.copyOf(children);
    }
    @Override public PolicyResult evaluate(BuyerContext b, CartContext c) {
        for (PurchasePolicy p : children) {
            PolicyResult r = p.evaluate(b, c);
            if (!r.isOk()) return r;     // short-circuit; report first failure
        }
        return PolicyResult.ok();
    }
}

public final class OrPolicy implements PurchasePolicy {
    private final List<PurchasePolicy> children;
    public OrPolicy(List<PurchasePolicy> children) {
        this.children = List.copyOf(children);
    }
    @Override public PolicyResult evaluate(BuyerContext b, CartContext c) {
        PolicyResult lastFailure = null;
        for (PurchasePolicy p : children) {
            PolicyResult r = p.evaluate(b, c);
            if (r.isOk()) return PolicyResult.ok();
            lastFailure = r;
        }
        return lastFailure;
    }
}

PolicyResult is a small VO with ok() and reject(message) factories — the message is what shows in the UI per req 3.5 ("Tickets are restricted to age 18 and over", not "request failed").

WHY Composite gives arbitrary nesting depth for free. The business rule "age ≥ 18 AND (qty ≤ 2 OR qty ≥ 100)" parses straight into an AndPolicy(AgeAtLeast(18), OrPolicy(QuantityAtMost(2), QuantityAtLeast(100))). No special-casing for nesting levels.

5.4 Polymorphic InventoryZone

V2 split InventoryZone into an abstract class with two subclasses. The two zone types have asymmetric internal state (standing has a counter; seated has a per-seat map) but share the same reserve / release / confirmSale contract.

Domain — InventoryZone hierarchy
public abstract class InventoryZone {
    protected final String zoneId;
    protected final String name;
    protected final BigDecimal price;
    protected final ZoneType type;

    protected InventoryZone(String zoneId, String name, BigDecimal price, ZoneType type) {
        this.zoneId = zoneId;
        this.name = name;
        this.price = price;
        this.type = type;
    }

    public abstract int availableCount();
    public abstract void reserve(InventorySelection selection);
    public abstract void release(InventorySelection selection);
    public abstract void confirmSale(InventorySelection selection);
}

public final class StandingZone extends InventoryZone {
    private final int capacity;
    private int reservedAmount;
    private final Object lock = new Object();

    @Override public int availableCount() {
        synchronized (lock) { return capacity - reservedAmount; }
    }

    @Override public void reserve(InventorySelection sel) {
        int qty = sel.requireStanding();
        synchronized (lock) {
            if (reservedAmount + qty > capacity)
                throw new IllegalStateException("Not enough capacity in zone " + zoneId);
            reservedAmount += qty;
        }
    }
    // release(), confirmSale() symmetric
}

public final class SeatedZone extends InventoryZone {
    private final Map<String, Seat> seats;                            // label -> Seat
    private final Map<String, ReentrantLock> seatLocks;               // label -> per-seat lock

    @Override public void reserve(InventorySelection sel) {
        List<String> labels = new ArrayList<>(sel.requireSeated());
        Collections.sort(labels);          // sorted order prevents deadlock
        List<ReentrantLock> held = new ArrayList<>();
        try {
            for (String label : labels) {
                ReentrantLock l = seatLocks.get(label);
                if (l == null) throw new IllegalArgumentException("Unknown seat " + label);
                l.lock();
                held.add(l);
                if (seats.get(label).status() != SeatStatus.AVAILABLE)
                    throw new IllegalStateException("Seat " + label + " is not available");
            }
            for (String label : labels) seats.get(label).markReserved();
        } catch (RuntimeException e) {
            for (ReentrantLock l : held) l.unlock();
            throw e;
        }
        for (ReentrantLock l : held) l.unlock();
    }
}
WHY Asymmetry is correct. Standing has no seats to enumerate; a counter is the right model. Seated has named, coordinate-bearing seats that need individual locking. Don't try to force them into a single shape. Lecture-style polymorphism: callers see InventoryZone; the subclass does the right thing.
PITFALLLock ordering matters. In SeatedZone.reserve we sort labels before locking. If two users reserve {"A1","A2"} and {"A2","A1"} simultaneously, unsorted locking can deadlock. Sorted lexicographic order guarantees a global acquisition order, eliminating the cycle.

5.5 Enums for state machines

State machines are «Enum» boxes connected to the entity in the diagram, and Java enum types in code.

6.2 Never return domain objects HARD RULE

Services return primitives or DTOs. Never a User, Event, or any aggregate-internal entity. DTOs live in Core/Application/dto/.

WHY Encapsulation. Once a User escapes a service, callers can mutate it directly, skipping every invariant check and lock your domain code enforces. The DTO is a snapshot, not a handle.
Application — DTO mapping

6.3 Transactional unit of work

Each service method is a single transactional unit — all or nothing. If anything throws mid-flight, the caller sees a clean failure; partial state never leaks. With Memory*Repository today, "transactional" means we use lockForUpdate + try / finally unlock. With JPA in V3 it'll mean @Transactional.

6.5 What services should NOT do

  • ❌ Hold business logic that belongs in an entity
  • ❌ Return domain objects
  • ❌ Hold mutable state across calls (services are stateless beans)
  • ❌ Call Vaadin or REST APIs (lower layers don't reach up)
  • ❌ Catch exceptions to hide bugs ("just log and continue" — no)

7. Infrastructure layer conventions

7.1 The IRepository<T, ID> contract

Every aggregate's IXxxRepository extends a shared IRepository<T, ID>. The shared interface gives every repository the same baseline contract — find, save, locking — without duplicating method signatures.

Domain — IRepository<T, ID>
package Core.Domain.shared;

import java.util.Collection;
import java.util.Optional;

/**
 * Shared parent contract for every aggregate-root repository in the system.
 *
 * Two responsibilities:
 *  - CRUD (find/save/findAll/delete)
 *  - Concurrency primitive (lockForUpdate/unlock) so callers don't need to
 *    know which sync mechanism the implementation uses.
 *
 * For V2 (in-memory): backed by a per-key ReentrantLock map (RepositoryLocks).
 * For V3 (JPA): will map to SELECT ... FOR UPDATE at the DB level.
 */
public interface IRepository<T, ID> {
    Optional<T> findById(ID id);
    Collection<T> findAll();
    T save(T entity);
    void deleteById(ID id);

    /** Acquire an exclusive lock on the aggregate with this id. Blocks. */
    void lockForUpdate(ID id);

    /** Release the lock acquired by lockForUpdate. Always pair via try/finally. */
    void unlock(ID id);
}

7.2 Memory*Repository implementation

Every memory repo follows the same shape: a ConcurrentHashMap keyed by ID for storage, a RepositoryLocks instance for the locking primitive, and an AtomicInteger for ID generation.

Infrastructure — MemoryUserRepository.java
package Infrastructure.persistence;

import Core.Domain.users.IUserRepository;
import Core.Domain.users.User;
import org.springframework.stereotype.Repository;

import java.util.*;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicInteger;

@Repository
public class MemoryUserRepository implements IUserRepository {

    private final Map<Integer, User> store = new ConcurrentHashMap<>();
    private final RepositoryLocks<Integer> locks = new RepositoryLocks<>();
    private final AtomicInteger nextId = new AtomicInteger(1);

    @Override
    public Optional<User> findById(Integer id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public Collection<User> findAll() {
        return Collections.unmodifiableCollection(store.values());
    }

    @Override
    public User save(User entity) {
        Objects.requireNonNull(entity, "user");
        if (entity.id() == 0) entity.assignId(nextId.getAndIncrement());
        entity.checkInvariants();          // belt + braces; in-memory persistence has no DB constraints
        store.put(entity.id(), entity);
        return entity;
    }

    @Override
    public void deleteById(Integer id) { store.remove(id); }

    @Override
    public void lockForUpdate(Integer id) { locks.lockForUpdate(id); }

    @Override
    public void unlock(Integer id) { locks.unlock(id); }

    @Override
    public Optional<User> findByUsername(String username) {
        return store.values().stream()
            .filter(u -> u.username().equalsIgnoreCase(username))
            .findFirst();
    }
}

7.3 RepositoryLocks — per-key locking primitive

Infrastructure — RepositoryLocks.java
package Infrastructure.persistence;

import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.locks.ReentrantLock;

/**
 * Per-key lock registry. Used by every Memory*Repository to provide the
 * lockForUpdate/unlock primitive in the IRepository contract.
 *
 * V3: this class disappears - JPA's pessimistic-lock takes over.
 */
public final class RepositoryLocks<ID> {

    private final ConcurrentHashMap<ID, ReentrantLock> locks = new ConcurrentHashMap<>();

    public void lockForUpdate(ID id) {
        locks.computeIfAbsent(id, k -> new ReentrantLock()).lock();
    }

    public void unlock(ID id) {
        ReentrantLock lock = locks.get(id);
        if (lock == null) return;          // never locked, nothing to do
        if (lock.isHeldByCurrentThread()) lock.unlock();
    }
}
WHY One lock per aggregate ID. Locking the entire store map would serialize all writes; locking nothing would race on concurrent updates of the same aggregate. Per-ID locking is the sweet spot — two threads working on different aggregates don't contend. ReentrantLock (not synchronized) so the same thread can re-enter without deadlocking itself.

9. Concurrency & locking

Multiple users hit the same resources at once — two people picking the same seat, two admins editing the same company, etc. The system must stay consistent under contention.

LECTURELecture 1, SLR §1: "All correctness constraints and integrity rules in the requirements document are preserved after every operation. The implementation must handle race conditions arising from concurrent users."

9.1 The lockForUpdate / unlock pattern

The repository contract gives every service a single way to serialize operations on a specific aggregate. Always pair lock with unlock in try / finally.

Application — ReservationService.reserve
public ReservationResultDTO reserve(int eventId, int userId, InventorySelection selection) {
    eventRepo.lockForUpdate(eventId);            // serialize on this event
    try {
        Event event = eventRepo.findById(eventId)
            .orElseThrow(() -> new EntityNotFoundException("Event " + eventId));
        event.reserveFor(userId, selection);     // domain method throws if invalid
        eventRepo.save(event);
        return new ReservationResultDTO(event.id(), selection, expiryTime());
    } finally {
        eventRepo.unlock(eventId);               // ALWAYS unlock
    }
}
PITFALLAlways use try/finally. If the domain throws inside the lock and you don't unlock, the next caller waits forever. finally guarantees the unlock runs even on exception.

9.2 Multi-aggregate ops — lock in stable order

Some use cases touch two aggregates (checkout = ActiveOrder + Event + Ticket + OrderReceipt). Acquire locks in ID order — same as the seat ordering in SeatedZone. Never lock A then B from one code path and B then A from another.

9.3 Per-seat locks inside SeatedZone

The seat picker reserves multiple seats atomically. Each seat has its own ReentrantLock; acquisition is in sorted label order to prevent deadlock between two users picking overlapping seats.

9.4 What NOT to lock

  • ❌ Don't lock on Ticket. Tickets are post-purchase; contention is negligible. The reservation path locks the Event, not the Ticket.
  • ❌ Don't lock on Session. Sessions are per-user; the user can only act on one session at a time anyway.
  • ❌ Don't synchronized on aggregate methods. That locks at the JVM object level; can't be re-implemented in V3 with JPA.

9.5 Concurrency tests use CountDownLatch

Lecture 1 made this explicit: regular tests aren't enough. To exercise race conditions, spin up multiple threads waiting on a starting line, then release them simultaneously.

Test — concurrent seat reservation
@Test
void twoUsersCantReserveTheSameSeat() throws Exception {
    Event event = makeEventWithSeatedZone("A", 1);   // single seat "A1"
    eventRepo.save(event);
    int eventId = event.id();
    String seat = "A1";

    int threads = 10;
    ExecutorService pool = Executors.newFixedThreadPool(threads);
    CountDownLatch startLine = new CountDownLatch(1);
    CountDownLatch done = new CountDownLatch(threads);
    AtomicInteger successes = new AtomicInteger();

    for (int i = 0; i < threads; i++) {
        int userId = i + 1;
        pool.submit(() -> {
            try {
                startLine.await();
                reservationService.reserve(eventId, userId, InventorySelection.seated(List.of(seat)));
                successes.incrementAndGet();
            } catch (Exception ignored) {
                // expected for 9 of 10
            } finally {
                done.countDown();
            }
        });
    }

    startLine.countDown();                   // GO
    done.await(5, TimeUnit.SECONDS);
    pool.shutdownNow();

    assertEquals(1, successes.get(), "Exactly one user should win");
}
WHY The latch-based pattern stresses the lock. Without the starting line, threads would queue up sequentially and the test would pass even with a broken lock. The simultaneous release maximises contention.

10. Logging — the AOP aspect

You don't write entry / exit log lines by hand. LoggingAspect wraps every public method in Core.Application.services..* via Spring AOP @Around advice. Add a service method → it gets logged for free.

10.1 What LoggingAspect does for free

Infrastructure — LoggingAspect.java (excerpt)
@Aspect
@Component
@Slf4j
public class LoggingAspect {

    @Around("execution(public * Core.Application.services..*.*(..))")
    public Object logServiceCall(ProceedingJoinPoint pjp) throws Throwable {
        String signature = pjp.getSignature().toShortString();
        int argCount = pjp.getArgs().length;
        long t0 = System.nanoTime();
        log.debug("→ {} ({} args)", signature, argCount);     // entry; no PII
        try {
            Object result = pjp.proceed();
            long ms = (System.nanoTime() - t0) / 1_000_000;
            log.debug("← {} ok ({}ms)", signature, ms);
            return result;
        } catch (Throwable e) {
            long ms = (System.nanoTime() - t0) / 1_000_000;
            log.warn("✗ {} threw {} ({}ms): {}", signature, e.getClass().getSimpleName(), ms, e.getMessage());
            throw e;
        }
    }
}
WHY Cross-cutting concerns belong in aspects. Without AOP you'd need log.debug("entering register with...") at the top of every service method and similar at every exit point. With AOP it's declarative — the aspect's pointcut expression covers the whole layer. Lecture 1: SLR §8 requires an event log and an error log; this is the skeleton for both.

10.2 @Slf4j on classes that need manual logs

Use Lombok's @Slf4j to get a static log field. Only add manual log calls where the aspect can't help — inside domain logic, inside infrastructure adapters, at lifecycle boundaries.

Negative scenarios are not errors (lecture 1) — a user typing a wrong password is not a system error and should log at INFO/DEBUG, not ERROR.

11. Testing — the BaseDomainTest harness

Lecture 1's testing rules: acceptance tests through Application only; mocks where appropriate; concurrency via CountDownLatch; negative tests are first-class. PR #201 added the BaseDomainTest harness so invariants are checked automatically after every test.

11.1 The harness

support/BaseDomainTest.java (test scope)
package support;

import Core.Domain.shared.InvariantChecked;
import org.junit.jupiter.api.AfterEach;

import java.util.ArrayList;
import java.util.List;

/**
 * Parent class for every domain / acceptance test. After each test method,
 * checkInvariants() is called on every aggregate the test "tracked", so
 * existing operations that quietly break invariants are caught.
 */
public abstract class BaseDomainTest {

    private final List<InvariantChecked> tracked = new ArrayList<>();

    protected <T extends InvariantChecked> T track(T entity) {
        tracked.add(entity);
        return entity;
    }

    @AfterEach
    void checkAllInvariants() {
        for (InvariantChecked entity : tracked) {
            entity.checkInvariants();
        }
    }
}

11.2 The track() pattern

Wrap aggregate creation in track(...). After the test method finishes, the harness calls checkInvariants() on each tracked aggregate. If a test mutated it into an invalid state, the assertion fails.

Example — User test extends BaseDomainTest
class UserTest extends BaseDomainTest {

    @Test
    void promotingGuestPreservesEmailFormat() {
        User user = track(new User(1, "alice", "hashed", "alice@example.com"));
        user.changeEmail("ALICE@example.com");        // operation under test
        assertEquals("ALICE@example.com", user.email());
        // @AfterEach calls user.checkInvariants() automatically.
        // If changeEmail accidentally blanked the email, the test would fail.
    }
}
WHY Catches the bug the explicit test forgot. An isolated "does checkInvariants throw on a hand-crafted invalid object" test only validates the check method. The point is to verify that real operations in the codebase preserve invariants under varied scenarios. Piggy-backing the check on existing tests is free coverage.
BADDon't create isolated XInvariantTest files. Adding UserInvariantTest with @Test void checkInvariants_throws_on_blank_username() only proves the check method works. It tells you nothing about whether AuthenticationService.register or MemberAccountService.updateProfile preserve invariants. Track inside existing tests instead.

11.3 Acceptance tests go through the application layer only

Lecture 1: acceptance tests don't reach into domain internals.

// GOOD - acceptance test enters via the service.
@Test
void aMemberCanRegisterAndImmediatelyBrowse() {
    var reg = authService.register(new RegistrationRequestDTO("alice", "P@ssw0rd!", "alice@example.com"));
    var browse = catalogService.browse(BrowseFilterDTO.all());
    assertNotNull(reg.userId());
    assertFalse(browse.isEmpty());
}

// BAD - reaches into domain internals; would belong in a UserTest.
@Test

    authService.register(...);
    User u = userRepo.findByUsername("alice").orElseThrow();
    assertTrue(u.passwordHash().startsWith("$2a$"));     // implementation detail
}

11.4 Mock framework — when to mock

  • Do mock external ports (IPaymentGateway, ITicketIssuer, INotifier) — never hit a real payment provider in tests.
  • Don't mock repositories. Use real Memory*Repository — they're already in-memory; mocking adds noise without speed.
  • Don't mock the domain. Use real aggregates with real invariants.

11.5 Negative tests are first-class

A test "the system rejects an under-age buyer" is just as important as "an adult buys successfully." Lecture 1 underlined this. Use assertThrows:

@Test
void minorCannotBuyAdultOnlyEvent() {
    PurchasePolicy adultsOnly = new AgeAtLeast(18);
    var minor = new BuyerContext(/*age*/ 17);
    var cart = new CartContext(2);
    PolicyResult result = adultsOnly.evaluate(minor, cart);
    assertFalse(result.isOk());
    assertTrue(result.message().contains("18 and over"));

11.6 What NOT to write tests for

  • ❌ Vaadin UI rendering — V2 spec explicitly says UI tests aren't required, only Domain + Application.
  • ❌ Spring framework behaviour — trust that @Service and @Autowired work.
  • ❌ Logging output. Aspect's job, not your test's.

15. Anti-patterns — things to NOT do

Compiled from lecture material + PR reviews on this codebase. Each one is a real mistake we've caught.

15.1 Returning domain objects from services

BAD 
public User getCurrentUser(String token) { return userRepo.findById(sessionToUserId(token)).orElseThrow(); }

Once User escapes, callers can mutate it, skipping every domain invariant. Always wrap in a DTO.

15.2 Locking on Ticket for the reservation path

Ticket is post-purchase. The contention is on the Event aggregate's zones, not on tickets. Lock the Event, not the Ticket.

15.3 Real-time push code in Tier C

Don't add @Push, Broadcaster, or WebSocket config. I.5 is exempt. The notification bell uses the offline path only.

15.4 Business logic in Views

BAD 
// In CartView's submit listener:
submit.addClickListener(e -> {
    if (cart.totalPrice().compareTo(BigDecimal.valueOf(100)) < 0) {
        Notification.show("Minimum order is $100");
        return;
    }
    if (user.age() < 18) {
        Notification.show("Adults only");
        return;
    }
    checkoutService.checkout(cart);
});

That's a purchase policy. It belongs on PurchasePolicy, not in the view. The view calls the service; the service applies the policy; the failure message comes back through the presenter.

15.5 Skipping checkInvariants() in tests

Always track(...) aggregates you build in tests. Catching invariant violations after a normal scenario is free coverage.

15.6 Using a Mockito mock for a repository

Memory*Repository is already a real in-memory implementation. Mocking it adds expectation maintenance for zero perf gain. Use the real one.

15.7 @Autowired field injection

Constructor injection only. Makes dependencies explicit, supports immutability, and lets you instantiate the class without Spring in tests.

15.8 Logging PII

Never log raw password, full email, card numbers, or session tokens. Counts + IDs + masked values only.

15.9 Cross-aggregate object references

Ticket.holder: User — no. Ticket.holderUserId: int — yes. Cross-aggregate refs are by ID only.

15.10 "Anemic domain" — God Services

If a service method is more than ~30 lines and full of ifs and field gets/sets on a domain object, you've pulled logic out of the domain. Push it back. The aggregate should expose operations, not setters.

15.11 Adding to the docs/ folder from any branch other than the docs owner's

The docs/ folder is only writable from Adam's environment. If your work involves writing docs, draft the content and hand it off — V2-MODEL-01, V2-UC-01, V2-DOC-01, V2-DOC-04 all live in Adam's lane for this reason.

15.12 Committing with an AI-attribution trailer

No Co-Authored-By: Claude, no Generated-by: footers. Commits look like solo student work in this repo.
