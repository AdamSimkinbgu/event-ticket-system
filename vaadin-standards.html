Why Vaadin, and how we use it

Vaadin Flow lets us write the web UI in Java, in the same project as our services and domain. No separate React app, no REST-API plumbing between us and our own code. We get a full set of pre-built UI components (buttons, grids, forms, dialogs) and we compose them like Lego.

The trade-offs you should know up front:

  • State lives on the server. Each browser tab holds a Java object tree on the server (the View). Don't put large in-memory collections in view fields.
  • The look is opinionated. The default Lumo theme is clean and accessible. We tweak with CSS where needed but we don't fight it.
  • No JavaScript needed. If you find yourself wanting JS, you're almost certainly doing it wrong — ask Adam or Naim first.
The lecturer's rule (from the V2 lecture): Vaadin uses the MVP pattern (Model-View-Presenter). Don't use MVVM with Vaadin. Don't use MVC the way React does it. Stick to MVP.

MVP — the architectural rule

One sentence: View calls Presenter. Presenter calls Service. View never calls Service directly.

The dependency chain. Presenter is the only thing that knows about both.

Why this matters

  • Testability. Presenters are plain Java objects — you can unit-test them without spinning up Vaadin.
  • Reusability. Two views can share a presenter (rare), and a presenter can be reused if we ever expose a REST endpoint that needs the same orchestration.
  • Clarity. When something goes wrong, you know where to look: UI bug → View; business logic bug → Service; "Why is this method being called?" → Presenter.

The minimum viable triple

// FooView.java
@Route(value = "foo", layout = MainLayout.class)
public class FooView extends VerticalLayout {
    public FooView(FooPresenter presenter) {
        Button btn = new Button("Do the thing", e -> presenter.doTheThing());
        add(btn);
    }
}

// FooPresenter.java
@Component
@Scope("prototype")  // one presenter per view instance
public class FooPresenter {
    private final FooService service;
    public FooPresenter(FooService service) { this.service = service; }
    public void doTheThing() {
        try { service.doIt(); Toasts.success("Done!"); }
        catch (RuntimeException ex) { Toasts.failure(ex.getMessage()); }
    }
}

Your first view (10 minutes)

The checklist for adding a new screen to the app:

  1. Create FooView.java in src/main/java/com/ticketing/system/Presentation/views/. Annotate it: @Route(value = "foo", layout = MainLayout.class).
  2. Extend VerticalLayout (default) or HorizontalLayout if the page is naturally side-by-side.
  3. Inject the presenter via the constructor.
  4. Build the UI in the constructor: add(new H2("Title"));, add(grid);, etc.
  5. Add the route to MainLayout's nav if it's a top-level page.
  6. Run ./mvnw spring-boot:run, open localhost:8080/foo, you should see your page rendered inside the app shell.
If you can do this in 10 minutes, you're ready. If you're stuck at the route or the security gate, see the App shell section below.

Border radius & shadows

  • Cards / panels: 12px radius, subtle shadow 0 1px 3px rgba(0,0,0,0.04)
  • Buttons / chips: 6px radius for chunky controls, full pill (999px) for tags
  • Input fields: use Vaadin defaults (don't override)

App shell — MainLayout & AdminLayout

Every view sits inside MainLayout. Admin views additionally sit inside AdminLayout, which is a sub-layout that adds a side nav for admin sections.

MainLayout — the shared frame

MainLayout — what every page sits inside. Top bar is fixed; route content fills the card below.

AdminLayout — sub-layout for admin pages

AdminLayout — wraps admin routes. Side nav lists admin sections, content renders on the right.

Route declaration

// Standard view
@Route(value = "my-cart", layout = MainLayout.class)
@PageTitle("My Cart")
@PermitAll
public class CartView extends VerticalLayout { ... }

// Admin view
@Route(value = "admin/history", layout = AdminLayout.class)
@PageTitle("Global history")

public class GlobalHistoryView extends VerticalLayout { ... }

// Anonymous (login / register / browse)
@Route(value = "login", layout = MainLayout.class)
@AnonymousAllowed
public class LoginView extends VerticalLayout { ... }

Layout components

The frames that hold everything else. Start here when building a page.

VerticalLayout DEFAULT USE

Stacks children vertically. The default container for any view.

Use whenYour content reads top to bottom: hero → search → list → footer.
Don't use whenYou need columns. Use HorizontalLayout or Split for that.
VerticalLayout v = new VerticalLayout();
v.add(new H2("Events"));
v.add(searchBar);
v.add(grid);
v.setSpacing(true);
v.setPadding(true);
v.setSizeFull();
HorizontalLayout USE

Lays out children left to right.

Use whenAction row (Cancel | Save buttons), inline icons + labels, top-bar layout.
Don't use whenYou're building a real two-column page. Use Split so the user can resize.
HorizontalLayout actions = new HorizontalLayout(cancelBtn, saveBtn);
actions.setJustifyContentMode(JustifyContentMode.END);  // right-align
FormLayout USE

Two-column responsive form layout. Label on left, input on right (collapses to one column on narrow screens).

Use whenAny form with 3+ fields — register, profile, event edit, policy editor.
Don't use whenYou have one field (overkill — just stack with VerticalLayout).
FormLayout form = new FormLayout();
form.addFormItem(usernameField, "Username");
form.addFormItem(emailField,    "Email");
form.addFormItem(passwordField, "Password");
SplitLayout USE

Two-pane layout with a draggable divider. Built for master/detail pages.

Use whenList on the left + detail on the right (event browse → details, admin org tree → node info).
Don't use whenThe two panes aren't related — use sibling cards instead.
SplitLayout split = new SplitLayout();
split.addToPrimary(eventList);
split.addToSecondary(eventDetail);
split.setSplitterPosition(35);  // 35% / 65%
Scroller USE

A box that scrolls its content when it overflows.

Use whenYou have a sidebar or panel that might overflow vertically.
Don't use whenThe whole page can scroll (just let the browser do it).
Accordion / Details USE WITH CARE

Collapsible sections. Useful for FAQ-style content; avoid for primary nav.

Input & form components

TextField USE

Single-line text input. The bread and butter.

Use whenNames, titles, emails, search bars, single-line user input.
Don't use whenThe user might type a paragraph — use TextArea.
TextField username = new TextField("Username");
username.setRequired(true);
username.setMinLength(3);
username.setHelperText("3–32 letters, no spaces");
PasswordField USE

Same as TextField but masks the value with dots. Always use this for passwords (V2 req 2 — never show plain text).

EmailField USE

TextField with built-in email-format validation. Use whenever the field is an email.

NumberField / IntegerField USE

Number input with up/down steppers. Use for quantity selection, prices, ages.

IntegerField qty = new IntegerField("Quantity");
qty.setMin(1); qty.setMax(10);
qty.setStepButtonsVisible(true);
qty.setValue(1);
TextArea USE

Multi-line text. Use for announcements body, complaint text, event descriptions.

ComboBox <T> USE

Type-ahead dropdown. Use when there are many options (event, company, user picker).

ComboBox<EventDTO> eventPicker = new ComboBox<>("Event");
eventPicker.setItems(eventService.list());
eventPicker.setItemLabelGenerator(EventDTO::name);
Select <T> USE

Plain dropdown, no search. Use when there are 3–10 options (category, status).

RadioButtonGroup <T> USE

Pick one of a small set. Use for zone-type selection (Standing / Seated), policy operator (AND / OR).

CheckboxGroup <T> USE

Pick many. Use for permission matrix (manager permissions).

CheckboxGroup<Permission> perms = new CheckboxGroup<>("Permissions");
perms.setItems(Permission.values());
perms.setItemLabelGenerator(Permission::name);
DatePicker / DateTimePicker USE

Calendar pop-up. Use for event dates, filter date ranges, suspension end dates.

Upload SKIP for V2

File-upload widget. Out of scope — we don't upload images, attachments, or files in V2.

Binder — data binding

Wire your form to a DTO with Binder<T>. It runs validation, shows error messages, and writes back to the DTO when the user saves. 

Binder<RegisterRequestDTO> binder = new Binder<>(RegisterRequestDTO.class);

binder.forField(usernameField)
      .asRequired("Username is required")
      .withValidator(StringLengthValidator("3–32 chars", 3, 32))
      .bind(RegisterRequestDTO::username, RegisterRequestDTO::setUsername);

binder.forField(emailField)
      .asRequired("Email is required")
      .withValidator(new EmailValidator("Not a valid email"))
      .bind(RegisterRequestDTO::email, RegisterRequestDTO::setEmail);

// On submit:
if (binder.writeBeanIfValid(dto)) {
    presenter.register(dto);
}

Data display components

Grid <T> MOST USED USE

A virtualised data table with sortable columns, filtering, pagination. Use for nearly every "list of things" — events, tickets, orders, announcements, managers, suspensions. 

Grid<EventSummaryDTO> grid = new Grid<>(EventSummaryDTO.class);
grid.setColumns("name", "date", "category", "location");
grid.addColumn(e -> "$" + e.minPrice() + "–$" + e.maxPrice()).setHeader("Price");
grid.addComponentColumn(e -> new Button("Reserve",
        click -> UI.getCurrent().navigate(EventDetailsView.class, e.id())));
grid.setItems(catalogService.search(filters));
TreeGrid <T> USE

Hierarchical grid with expand/collapse rows. Use for the Organisational Tree (UC-25, V2-VIEW-NW-02) and the policy editor's composite tree. 

TreeGrid<OrgNode> tree = new TreeGrid<>();
tree.addHierarchyColumn(OrgNode::name).setHeader("Name");
tree.addColumn(OrgNode::role).setHeader("Role");
tree.setItems(rootNodes, OrgNode::children);
VirtualList <T> USE WITH CARE

Like Grid but each row is a free-form component. Use only when Grid doesn't fit (e.g., a feed of styled notification cards).

Avatar / AvatarGroup USE

Round profile pictures with fallback initials. Use in the top bar for the current user and in conversation lists.

Badge / Tag USE

Small coloured labels. Use for status (ON_SALE, SOLD_OUT, REFUNDED), role chips, "3 new" badges. Apply Lumo theme variants.

Span badge = new Span("ON SALE");
badge.getElement().getThemeList().add("badge success");

Action & navigation

Button USE

The thing the user clicks. Pick the right variant:

VariantWhen
LUMO_PRIMARYThe main action on the page (Save, Reserve, Pay). One per page max.
(default)Secondary actions (Cancel, Filter, Open Dialog).
LUMO_TERTIARYInline action (edit pencil, delete trash) — text-only, no border.
LUMO_ERRORDestructive (Delete, Cancel Event). Always pair with a confirmation dialog.
LUMO_SUCCESSRare — explicit positive confirm (Approve).
Button reserve = new Button("Reserve", e -> presenter.reserve());
reserve.addThemeVariants(ButtonVariant.LUMO_PRIMARY);
MenuBar USE

Top-level menus with dropdowns. Use in the user-menu corner of MainLayout.

ContextMenu USE

Right-click menu attached to a component. Use on grid rows for per-row actions.

RouterLink USE

Vaadin's <a href> equivalent — navigates without a full page reload.

new RouterLink("My Cart", CartView.class);
Tabs / TabSheet USE

Horizontal tabs. Use to slice a dense detail page (Event Details: Overview | Map | Reviews | Policies).

SideNav USE

Vertical nav with collapsible sections. Use inside AdminLayout.

Feedback components

Notification (toast) REQ 3.5 USE

Short-lived banner at the bottom/top. Use for success and error messages — wrap in our Toasts helper so every view uses the same theme. 

// In Toasts.java (utility class)
public class Toasts {
    public static void success(String msg) {
        Notification n = Notification.show(msg, 4000, Position.TOP_END);
        n.addThemeVariants(NotificationVariant.LUMO_SUCCESS);
    }
    public static void failure(String msg) {
        Notification n = Notification.show(msg, 6000, Position.TOP_END);
        n.addThemeVariants(NotificationVariant.LUMO_ERROR);
    }
}
Dialog USE

Modal pop-up that blocks the page. Use for forms that don't deserve their own route (Manager invitation), confirms (Cancel event), single-step wizards.

Dialog dlg = new Dialog();
dlg.setHeaderTitle("Cancel event?");
dlg.add(new Span("This will refund 47 tickets totalling $4,700."));
Button confirm = new Button("Cancel event", e -> { presenter.cancel(); dlg.close(); });
confirm.addThemeVariants(ButtonVariant.LUMO_ERROR);
dlg.getFooter().add(new Button("Nevermind", e -> dlg.close()), confirm);
dlg.open();
ConfirmDialog USE

Pre-built confirmation dialog with Confirm / Cancel buttons. Use when you'd otherwise build your own with two buttons.

ProgressBar / ProgressBar(indeterminate) USE WITH CARE

Use the indeterminate variant during the checkout call (it takes a moment for payment + ticket issuance). Don't use it for sub-second operations.

Tooltip USE

Hover hint. Use for icon-only buttons (to disclose what they do) and for "What's this?" question marks.

CheckoutView — Mohamad, V2-CHECK-01

Global countdown banner on top (one timer for the whole order); two columns below — order summary (with coupon) on the left, payment form on the right.
Timer behaviour on entry to checkout. When the user clicks Check out in the cart, the system does two things atomically:
  1. Refresh every reservation TTL for the active order to the same new expiry (e.g. now + 10 minutes). The per-line timers from CartView no longer apply — they were superseded by the new uniform expiry.
  2. Display one global countdown at the top of CheckoutView. If it expires before payment completes, the system releases all reservations and routes the user back to /cart with a clear failure toast ("Your reservations expired before payment finished — please reserve again").
Mohamad's CheckoutService needs a prepareCheckout(activeOrderId) method that performs the TTL refresh and returns the new expiry timestamp. The CheckoutView calls it once on attach, then a 1-second poll updates the displayed countdown.

VenueMapEditorView — Abed, V2-CADMIN-01

Zone list on the left. Right pane: zone metadata form + visual seat grid editor for seated zones.

PolicyTreeEditor — Abed, V2-PEDIT-01

Composite pattern as a visual tree. Orange = composite (AND/OR), green = leaf rule.

Recipe 1 — Bind a form to a DTO

Binder<RegisterRequestDTO> binder = new Binder<>(RegisterRequestDTO.class);
binder.forField(usernameField)
      .asRequired("Username required")
      .withValidator(new StringLengthValidator("3–32 chars", 3, 32))

Recipe 4 — Toast helper (build this once)

public final class Toasts {
    private Toasts() {}
    public static void success(String msg) { toast(msg, NotificationVariant.LUMO_SUCCESS, 4000); }
    public static void failure(String msg) { toast(msg, NotificationVariant.LUMO_ERROR,   6000); }
    public static void warn(String msg)    { toast(msg, NotificationVariant.LUMO_WARNING, 5000); }
    private static void toast(String msg, NotificationVariant v, int ms) {
        Notification n = Notification.show(msg, ms, Position.TOP_END);
        n.addThemeVariants(v);
    }
}

Recipe 5 — Live countdown timer

// Inside CartView for the per-line expiry chip
Span countdown = new Span();
countdown.addClassName("countdown");
UI.getCurrent().setPollInterval(1000);  // 1-second poll
UI.getCurrent().addPollListener(e -> {
    Duration left = Duration.between(Instant.now(), line.expiresAt());
    countdown.setText(String.format("⏱ %d:%02d", left.toMinutes(), left.toSecondsPart()));
    if (left.toMinutes() < 2) countdown.getElement().getThemeList().add("warning");
});

Anti-patterns — don't do this

1. View calls Service directly. If your view has a @Autowired ReservationService, you skipped the presenter. Refactor: introduce a presenter that the view talks to.
2. Notification.show() raw, everywhere. Use Toasts.success / Toasts.failure instead so the theme and duration are consistent.
3. Hand-rolled HTML or CSS strings. If you find yourself writing setElement().setProperty("innerHTML", "..."), stop. Build it with components instead.
4. Storing the whole result set on the View. If your view holds List<EventDTO> all = catalogService.findAll() in a field, that list lives on the server per browser tab. Use the grid's lazy-loading callback instead: grid.setItems(catalogService::page).
5. Two PRIMARY buttons on one page. Pick one main action. Everything else is default or tertiary.
6. System.out.println for debugging. Use log.debug — Adam's AOP aspect already wires @Slf4j into every class.
7. Long-lived modal pop-up chains. If the user has to open a dialog inside a dialog inside a dialog, you're hiding a missing page. Build the real page.

Accessibility & security checklist

Accessibility (do)

  • Every input has a label (or accessible name on an icon-only button via setAriaLabel).
  • Every button reachable by keyboard (Tab + Enter). Vaadin defaults handle this; just don't override.
  • Colour contrast: never use accent orange on white as body text. Use it for borders and chips only.
  • Don't convey state by colour alone — pair it with text (e.g. "ON SALE" badge, not just a green dot).

Security (must)

  • Req 2: all password fields are PasswordField. No TextField for passwords ever.
  • Req 3.4: hide nav links and buttons users can't access. The server still enforces the rule — UI hiding is for clarity, not for security.
  • Req 3.5: every action shows a success or failure message. No silent failures.
  • Never put credentials, tokens, or PII in toast messages or query strings.
  • Trust no client input. Validate on the server (services / domain), not just on the binder.
